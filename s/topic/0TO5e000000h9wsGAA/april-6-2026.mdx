2. **Smarter Non-Maintenance Intake & Rule Processing (Beta)**

We’re introducing the first step in our **Generic Intake roadmap**, designed to better handle non-maintenance requests and improve how intake rules are evaluated—so the right action is taken every time.

**Non-Maintenance Intake Fallback**\
If a resident submits a request that does not match any existing Intake Settings rules, we will now **send an email notification** to the default Property Manager on the property (or a custom email of your choice) instead of creating a work order. This helps ensure non-maintenance requests are surfaced without cluttering your work order queue.

**Holistic Rule Processing (Beta)**\
Alongside this, we’re testing a smarter approach to intake logic with Intake Settings. Instead of processing rules in a top-down order, it will **evaluate all rules at once** and match requests to the **most relevant and specific rule**, improving routing accuracy and reducing misclassification.

<Check>
  **Why This Matters?**

  - Fewer unnecessary work orders for non-maintenance requests
  - More accurate routing based on intent, not rule order
  - Greater visibility into requests that previously may not have matched any rule
</Check>

4. **Smarter Weather-Based Intake Rules**

We’ve updated how weather data is evaluated within Intake Settings to improve accuracy and consistency when triggering rules.

<Check>
  **What's New:**

  Intake Settings will now **reference the next 72 hours of weather data by default** when evaluating rules. This ensures your rules have a broader and more reliable view of upcoming conditions—helping capture events like incoming storms or temperature drops before they happen.
</Check>

**More Control for Your Team**

Need a shorter window? You can customize this directly in your Intake Settings:

- Specify a **smaller timeframe (e.g., 24 hours)** within your rule logic
- Tailor rules based on how proactively you want to respond to weather events

### **Resolved Bugs**

**Service Provider Territoties Error**

We’ve resolved an issue where Property Managers encountered an “invalid request” error when adding a Service Provider with service territories. This error previously blocked provider creation from the PM side, even though Latchel could successfully add the same provider on their behalf. With this fix, Service Provider territories can now be added without issue, ensuring a smoother and more reliable vendor setup experience.

<Tip>
  **Did you know?**

  You can assign service provider territories (zip codes) to better control which vendors are contacted for work orders—helping reduce duplicate setup across portfolios and properties while ensuring the right providers are dispatched.
</Tip>
