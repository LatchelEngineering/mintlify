## Page Tile or Name

The tile of the page should be self-explanatory and intuitive. It should include the main keywords the team will use when searching for the page.

For documents supporting the team operations, or day-to-day activities for each role, we will name them to fit the categories below:

| Type of Content | Start Page Title with | Example |
| --- | --- | --- |
| Standard Operating Procedures (SOP), or documents explaining how to execute a certain process, task or activity | How to: | How to: Create a new Cadence in Salesflot |
| Tips or Best Practices on how to best execute certain tasks, i.e. resources to help with ideas, or providing guidance, or teaching certain skills, or with templates that could be used to guide on how to best approach a task, etc. | Job-Aid: | Job-Aid: Cold Calling Best Practices |
| Frequently Asked Questions (FAQ) - Although this is a type of Job-Aid, the FAQs are usually created to help the team to quickly find an answer to a common question, therefore, naming them as FAQ will help to bring them up to the top of the search. When creating FAQs, DO NOT repeat content from other pages, but link them to the specific spot in the resource that contains the answer | FAQ | FAQ: Property Manager Questions about Benefits |

You can also use emoticons to add some fun to the title!

## Text Boxes

<Note>
  **Blue Boxes**

  Are used to highlight some key information about the document/ resource
</Note>

<Tip>
  **Green Boxes**

  Are used to show / highlight / emphasize STANDARD verbiage or language that we should use as a team when communicating with customers or the outside world. For example, our vision, mission, log lines
</Tip>

<Danger>
  **Red Boxes**

  Use this type of box for critical actions or behaviors that need to be avoided (e.g. "Never provide a Property Manager's private contact information to the resident")
</Danger>

<Warning>
  **Yellow Box**

  Use this type of box to caution about doing something or for important reminders through out the process.
</Warning>

## General Guidelines on How to Write a SOP

Refer to this document to know and use the existing official Latchel terminology: [Latchel's Terms and Definitions](https://latchel.atlassian.net/wiki/spaces/LTG/pages/2545811472/Latchel+s+Terms+and+Definitions).

If needed, you can use "Find and Replace" mode by clicking "ctrl\+F" to update any outdated terms in your document.

- Use a numbered list to describe steps that need to be taken in a certain sequence.
- Use a bullet list to show a list of items that do not need to be described in an specific order.
- Don't combine multiple steps into a single bullet, each bullet needs to be a single step.
- Use indentation ("Tab" key) to list steps that fall within another process.
- Use quotation marks to explicitly mention options within the application (e.g. Click the "Take Action" button).
- Bold, italics, underline, text colors: are helpful when you want to draw the attention of the reader to an important point within regular text.

Refer to the following link for information on how to use the confluence editor toolbar: [The Editor | Confluence Data Center 10.2 | Atlassian Documentation](https://confluence.atlassian.com/doc/the-editor-251006017.html)

## Keep it digestible and maintainable

- Try to maintain the text in the SOP to a minimum so that the SOP is easy to consume. Long SOPs are difficult to digest and quickly forgotten.
- Images are also a good resource to explain clearly with minimum text.
- When using images, use a software tool that allows you to markup screenshots/images to focus the attention of the reader to the relevant part of the image.
- Some software tools are: "Paint" (included with Windows), "Google Slides", "Lightshot" (Google chrome extension), etc.
- If the process is explaining how to use a software or execute process steps in a software, it may be easier to record a video. But avoid explaining policies or other things outside of the software usage. Those should be in written to make it searchable
- Refrain from explaining guided workflows of the platform, unless the workflow can lead the agent to an incorrect action, explaining each step adds complexity to the SOP and makes SOP maintenance harder.
- Having to explain a guided workflow is an indication that the workflow itself is not intuitive and this needs to be fixed with product updates.
- Always refer to existing SOPs when an action needs to be performed that is already documented somewhere else. Do not duplicate information, this is important so that we have one single source of truth.

For example, see the following text extract and notice how it refers back to an existing document:

_"The following documents will explain how to assess an emergency situation and determine if emergency dispatch is required or if the emergency can be de-escalated. It is also important to check for any potential duplicates to avoid double dispatch of vendors and determine if there's potential for property damage._

_To determine if there's potential for property damage refer to: [↔ Job Aid: Defining Property Damage](https://latchel.atlassian.net/wiki/spaces/OPS/pages/745504771)"_

Linking to other Published SOPs can be easily done when you copy the URL of the Published SOP and paste into the document, it automatically becomes its own link.

This can also be done by linking to existing page by clicking "ctrl\+k".

## Document Review and Publishing

After creating a new page, it is automatically a "Draft" until you hit the "Publish" button on the upper right.

<Frame>
  ![Publish Button](/images/Sales-5.png)
</Frame>

If you need to save the changes without publishing, just click "Close"

It's best to review documents inside of confluence because that will allow you to see the final format.

Share the draft with any stakeholders for review by copying the URL of the draft.

Reviewers can add comments to the draft by highlighting the related text and clicking "comment"
