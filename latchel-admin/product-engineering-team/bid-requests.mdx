## What is the Product Release?

Property Managers can now ask multiple Service Providers to provide estimates for a (typically large) job request. This helps them feel confident that they are choosing the best Service Provider and price for the job. They will also be able to track these requests in a batched view.

## What was the Previous State?

Prior to this release, the process for asking for bids was very cumbersome and manual. They'd have to create individual WOs for each Service Provider they wanted to ask for a bid, manually assign them, and then ask each Service Provider to look for the "Add an Estimate" option within their public dashboard to submit their bid. The PM would also have to keep track of all the individual WOs so that they could compare the estimates to each other. This process is so cumbersome so many PMs just collected bids through email (outside of Latchel).

## Who Is Impacted?

The following groups are impacted:

- **PMs** - PMs will be the only user group to be able to create bids and see the new Bid Requests page.
- **Latchel Admins/Ops** - Latchel Admins/Ops will only see the individual bid request WOs. These WOs will be marked with \[Bid Request\] in the title and just go through the normal WO flow as normal severity WOs. If a PM needs help with their bid request page or creating a bid, Latchel Admins can see what they are seeing through impersonation.

## When Is It Relevant?

This feature is relevant for when a PM wants to ask one or more Service Providers for a bid request for a single job. They will also have the ability to track and manage the individual requests within a bid request after they've been sent out.

### Bid Request Work Order Lifecycle

See overall demo here.

<Frame>
  ![Bid request work order lifecycle diagram](/images/Eng-231.png)
</Frame>

Once a bid request is created, individual work orders are created and assigned to each of the Service Providers that have been chosen to provide a bid. While each of these work orders will be grouped as part of the overall bid request, they will individually go through the following lifecycle:

- **Scheduling** - Depending on what information is available, each WO will go through different states to schedule an appointment for collecting the bid. These states can be:
  - **Scheduling with Resident** - If the Resident has not provided any availability within Latchel.
  - **Scheduling with Service Provider** - This state requires the Service Provider to choose a time that works for them based on the Resident's availability.
- **Awaiting Work Date** - Once the Service Provider has scheduled the appointment time for providing a bid, the WO will go into awaiting work date.
- **Awaiting Estimate** - This is a NEW state that only applies to WOs that are part of an overall bid request. Once the scheduled appointment time has past for a WO that's part of an overall bid request, the WO will automatically convert to this new "Awaiting Estimate" state. This indicates that we are waiting on the Service Provider to provide an estimate.
- **Needs Approval** - Once the Service Provider has submitted their estimate, the WO will go into Needs Approval for a PM to approve or deny. PMs are welcome to wait for the other WOs to go into "Needs Approval" state before making a final decision.

As a PM receives WOs in the "Needs Approval" state, he/she can choose to:

- **Deny** - Denying an estimate for an individual WO within a bid request will only impact that WO and move it to "Awaiting Invoice" state. The reason for it going into this state is so that the Service Provider can be paid for the time/work done to create the bid (if applicable).
- **Approve** - Approving an estimate for an individual WO within a bid request does the following:
  - Moves that specific WO into scheduling so that the Service Provider can then schedule the work date to do the actual work for his/her approved bid.
  - Automatically denies or cancels the other WOs in that bid request. The action depends on what state the other WOs in the bid request are at. If the other WOs in the bid request are in the following states, there will be an automated action on them when a WO within the bid request is approved:
    - **Scheduling with Resident or Scheduling with Service Provider** - The WO will be cancelled.
    - **Awaiting Work Date** - The WO will be cancelled.
    - **Awaiting Estimate** - The WO will be moved to Awaiting Invoice so that the Service Provider can be paid for the time/work done to create the bid (if applicable).
    - **Needs Approval** - The estimate on this WO will be automatically denied and moved to Awaiting Invoice. There will be a log item specifying that the estimate was denied due to another WO within the bid request being approved.

## What Are Some Risks or Ways Things Could Go Wrong?

Because we kept the "Awaiting Work Date" state name the same, Service Providers could potentially miss that the WO is a request for a bid and instead end up doing the actual work. The way that we've tried to mitigate this was by putting \[Bid Request\] in the WO title and making the main action for these WOs to "Add an Estimate". As a result, we find that the risk is pretty minimal.

Note that this is definitely a P0 version of bid requests and we anticipate that we'll continually add features and iterate to support more use cases.
