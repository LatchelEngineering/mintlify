## Standard Ticket Format

### Where does this apply?

This process should be followed any time we're proposing net-new or significantly improving existing functionality in the Latchel Application

### Which issue type should I use?

If the initiative is literally a single user-story, it can be submitted as a story. If there are likely to be multiple stories, it should be submitted as an epic. Epics will be broken down into stories as part of the tech grooming process.

### Epic Format

| Ticket Section | Description | Example |
| --- | --- | --- |
| Problem/Opportunity | A clear, written definition of the problem. This must include (1) a description of where the problem starts and ends (2) a hypothesis of what customer benefits will be achieved by solving this problem. Desired outcomes may be proposed and documented but implementation details must not be included. | "PMs are in the tough position of having to balance their Owner's needs, who generally want LOTS of information about every maintenance spend on a property and the vendor's needs, who typically want to do the least work possible to create an invoice and get paid. The current macroeconomic situation gives vendors a lot of power because skilled labor is in short supply. PMs like our generated invoice template because it collects lots of information and requires the vendor to include pictures. They want to experiment with forcing vendors to use it but are hesitant to enforce this at a system level since it could result in revolt from vendors which would damage their maintenance operation in the short term. We also know we want to experiment with giving PMs more granular control of the types of invoice data they collect and how its enforced. We believe that by giving PMs more granular control of which types of invoices can be submitted for a given work order, we'll improve PMs availability to deliver unbeatable service and build trust in our vendor billing product brand which is important for Latchel Pay." |
| Core user-story | This is an overarching user-story that attempts to capture the "core value" of the initiative. Think of this as a generalized version of the more precise user stories. We'll use this to help align on the best and quickest way to deploy something that helps us validate the hypothesized value of the core value of the initiative | "As a PM I can designate which work orders require Latchel-generated invoices so that I can have more control of my maintenance billing process and to better satisfy owner needs." |
| User-stories | Short requirements or requests written from the perspective of an end user. Should follow the format "As a \[persona\], I \[want to\], \[so that\]." | "As a PM I can designate which vendors are forced to use Latchel-generated invoices" |
| Goal | A clear, written description of the initiative's goals. Goals should be clearly aligned with at least one of the roadmap key results. If it takes longer than an hour to determine what a metric target should be, half it or double it! | "Increase the percentage of preferred vendor invoices that leverage our generated invoice template from 26% to 40%" |
| Grooming Notes | Contextual details relevant to discussing the initiative. This might include a loom video or screenshots of any existing or related experiences in the application | [Vendor Invoice Collection Setting](https://www.loom.com/share/6217e973014b41f6821993a6283d8168) |
| Mock-ups | If a UI element is being impacted, visual guidance should be provided. This can be scrappy mock-ups created by PMs or fancier figma files created by the Design team. |  |

### Story Format

| Ticket Section | Description | Example |
| --- | --- | --- |
| Problem/Opportunity | A clear, written definition of the problem. If part of an epic, the epic problem can be referenced with a summary of the aspect/user-story this ticket addresses | "See epic for detailed problem description. For the first exploratory phase, we will enforce the type of invoice collected at the Vendor Level." |
| User-stories | Short requirements or requests written from the perspective of an end user. Should follow the format "As a \[persona\], I \[want to\], \[so that\]." | As a Property Manager, I can designate which preferred vendors are forced to use Latchel-generated invoices |
| Acceptance criteria | These are a list of testing criteria that the PM plans to test in relation to the user-stories. These should be written in general language following a Given, When, Then format. These may not be developed until the planning stage (when individual story tickets are created) since they rely on the solution product/engineering aligned on. Engineering may break these down into more technical definitions once the ticket is accepted into the engineering SDLC. | PM/Admin AC1: Given a PM/Admin user on the [https://app.latchel.com/admin/contractors/mgmt](https://app.latchel.com/admin/contractors/mgmt) page, and "preferred vendor settings" modal. When the user turns the toggle to "ON". THEN any future invoice submitted by the vendor for that vendor\_pm relationship should be a latchel generated invoice |

### Bug Format

| Ticket Section | Description | Example |
| --- | --- | --- |
| Title | The title should provide a quick description of the bug. | Certain properties not merging |
| Test Details (as applicable) | Environment: Which branch are you using? Tester: Who reported this bug? \*Slack Link: if reported through slack, include a link to the thread. \*Full Story Link: Include customer link to Full Story \*OS: Operating System name \*Browser: What internet browser are they on? \*If applicable | Environment: [app.latchel.com](https://app.latchel.com/admin/contractors/mgmt); Tester: Malen Macasaquit; [Slack Link](https://latchel.slack.com/archives/CQ1EJT880/p1652126417085449); Full Story Link: N/A; OS: Windows 10; Browser: Google Chrome |
| How to Reproduce? | 1. Video using Loom OR 1. Step-by-step instructions on how to reproduce the issue | Loom Link: [Latchel - 9 May 2022](https://www.loom.com/share/49a710c4812748b889062153a9c69f41) OR 1. [Visit this link](https://app.latchel.com/admin/pm-company/15812/properties) 2. Search for 2069 Belmont Circle 3. Select the option to show all disabled properties 4. Select both 2069 Belmont Circle properties. 5. Click the bulk update menu at the top right and select Merge. 6. Select the property created on May 12, 2021 and click Merge. |
| Expected Result | How is the software supposed to work, with regard to the particular area in which the bug appears? The developer needs to know what the requirement is, in order to gauge the extent to which the bug is disrupting the user experience. Describe the ideal end-user scenario, and try to offer as much detail as possible. Don't just leave it at "the app is crashing, and it shouldn't." | I expect when I select the merge option, then select a property. All the data from the unselected properties should be merged into the property I selected. I should be left with one unit and the rest are no longer searchable/visible. |
| Actual Result | Detail what the bug is actually doing, and how it is a distortion of the expected result. Are you getting any errors? Is the page crashing? Is it unresponsive? | When I click the merge option, it's unresponsive. I refreshed the page to make sure it wasn't just a UI issue. Nothing changed. |
| How to Reproduce? | Step-by-step instructions on how to reproduce the issue | 1. [Visit this link](https://app.latchel.com/admin/pm-company/15812/properties) 2. Search for 2069 Belmont Circle 3. Select the option to show all disabled properties 4. Select both 2069 Belmont Circle properties. 5. Click the bulk update menu at the top right and select Merge. 6. Select the property created on May 12, 2021 and click Merge. |
| \*Video Evidence | Provide a recording of the bug to pair with the written instructions on how to reproduce | Loom Link: [Latchel - 9 May 2022](https://www.loom.com/share/49a710c4812748b889062153a9c69f41) |
| Severity/Priority | Low: Bug can be fixed at a later date. Other, more serious bugs take priority. Medium: Bug can be fixed in the normal course of development and testing. High: Bug should be pulled into the sprint as soon as there is availability as it's causing poor app experiences. Highest: Bug must be resolved at the earliest as it affects the system adversely and renders it unusable until it is resolved. These are posted to #tech-escalations | Medium |
