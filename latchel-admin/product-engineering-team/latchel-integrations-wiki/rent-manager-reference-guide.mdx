## **What happens when a customer first activates their integration?**

When a customer first activates their integration, a full synchronization of all of their properties and residents is triggered automatically. We refer to this as the “**initial sync**.”

## **Properties**

| **Platform field** | **Field/Interpretation in Latchel** |
| :-- | :-- |
| Address 1 | address 1 |
| Address2 | address2 |
| City | city |
| State | state |
| Zip Code | zip |
| Platform property ID | ref\_property\_id |
| Platform unit ID | ref\_unit\_id |
| Active?\* | is\_active \*If property is “inactive” in the platform, the property will be marked as “inactive “in Latchel |
| (various data) | integration\_data |

## **Residents**

Residents are not sync at any point if:

- RM Contact is a lease guarantor
- RM Contact is less than 18 years old
- The lease is inactive

The following resident fields are brought over from RM to Latchel via the initial sync:

- ref\_user\_id
- name
- email (nullable)
- phone (use the first number available in RM)
- ref\_property\_id
- ref\_unit\_id (nullable)
- move\_in\_data (check if valid date object, if not, set to null)
- move\_out\_data (check if valid date object, if not, set to null)
- integration\_data
  - rm\_user\_id =\> tenant ID
  - rm\_contact\_id =\> contact ID

## **Work orders**

The following Work Order (Job) fields are brought over from RM via the Integration:

- ref\_work\_order\_id
- work order number
- name (WO Title)
- description
- scheduled\_data
- vendor\_id (Payee Account ID)
- state\_id (special logic applied to format to Latchel)
- is\_urgent (special logic applied to format to Latchel)
- is\_emergency =\> always set to 0
- ref\_property\_id
- ref\_unit\_id (nullable)
- ref\_user\_id (nullable)

Each work order is:

- Assigned to a property manager based on the pm\_user id related to the property
- Linked to an existing property & unit in Latchel
- IF a resident id is supplied, the WO will also be assigned to an existing resident in Latchel.

## **Leases**

Leases are not currently processed as part of the initial sync

## **What happens after the initial sync?**

There are two directions of data synchronization that occur after the initial sync.

1. Platform → Latchel
2. Latchel → Platform

Subsequent syncs will look for any data that was created or updated **within the last 72 hours.** The syncs run every hour, starting sometime in the first 5 minutes of the hour and may take between 15-45 minutes to complete.

## **Platform → Latchel**

### **Properties**

Subsequent property updates can be organized into two categories (1) Newly created properties and (2) previously created properties

#### **Newly created Properties**

Newly-created properties will sync all the same fields we do in the initial sync. [<u>See above!</u>](https://latchel.atlassian.net/wiki/spaces/PROD/pages/2618261539/Propertyware+Reference+Guide#Properties "https://latchel.atlassian.net/wiki/spaces/PROD/pages/2618261539/Propertyware+Reference+Guide#Properties")

#### **Previously created Properties**

Subsequent synchronizations of the same property will impact the following fields:

| :-- | :-- |
| Active? | is\_active (only from active to inactive) **Note**: This means if a customer _activates_ an _inactive property_ in the platform, **this change will not propagate to Latchel.** |
| (various data) | integration\_data |
| Address data\* | address\_label \*If any address data is different in the platform than in Latchel, we store the exact platform address in order to provide the “Source Address” feature for customers. [<u>Link to product release.</u>](https://latchel.slack.com/archives/C038Z6M0T6H/p1735848840482549 "https://latchel.slack.com/archives/C038Z6M0T6H/p1735848840482549") |

##### **Automatic Property Reconciliation**

To ensure accurate customer billing Latchel runs on automated process each month to de-activate any active units in Latchel that no longer exist in the Rent Manager property list integrated with Latchel. This process currently happens on the 27th at 3am PST.

Customers can view information about recently de-activated units on the Reports page. They can also export a list of the units deleted in the most recent snapshot.

When a property is deactivated through this process. It will be logged to the property in Latchel

![](/images/image-20240112-042855.png)

#### **Common areas**

Upon customer request, Latchel can create common area units for PM’s to manage building-level maintenance for jobs such as landscaping, garbage removal, elevator maintenance and more.

How it works:

1. A PM (or Latchel team member on behalf of a PM) submits a one-off request for common area maintenance on a specific building(s) via this Typeform: [https://latchel.typeform.com/to/P8jj3FBx](https://latchel.typeform.com/to/P8jj3FBx)
   1. Why one-off requests only? We only want to enable common areas on specific addresses for customers who are actually going to create building-level work orders due to billing implications (they will appear as active units in Latchel for billing purposes).
2. Latchel creates the common area unit via API.
   1. This results in a record with a building ID but no ref unit ID.

Subsequent resident updates can be organized into two categories (1) Newly-created residents and (2) previously created residents

Once a resident is created the only thing we only change a few things on subsequent syncs

- The property and unit the resident is associated with
- The move\_in and move\_out dates

The integration will not currently update contact information, name nor “active” status.

### **Work Orders**

Subsequent work order updates can be organized into two categories (1) Newly-created work orders and (2) previously created work orders. First, its important to understand the below requirement for WOs to sync from Rent Manager to Latchel.

#### **Sync Requirements**

If a WO is created in Rent Manager it **must** have the **Category** set to “**\[Latchel\] Maintenance”** for it to be synced over to Latchel.

![](/images/image-20230213-192920.png)

#### **Newly-created work orders**

Newly-created work orders will sync all the same fields we do in the initial sync.

#### **Previously-created work orders**

The integration will not update any fields on previously created work orders

### **Leases**

The lease sync runs at 11:30pm UTC every night and are thus synchronized from RM to Latchel once a day.

The following lease fields are brought over from RM to Latchel via the daily sync:

- field\_1
- field\_2

#### **Newly-created leases**

Sync’d daily at 11:30pm UTC every night

#### **Previously-created leases**

We update all values if it is an existing lease. So whatever new values come in from the sync would override the old ones.

## **Latchel → Platform**

### **Work orders**

#### **Newly created WOs**

If a work order (job) is created in the Latchel console it will automatically trigger a sync to also create it in RM.

- `ServiceManagerIssueID (ref_job_id)`
- `Title` (name)
- work order number
- `IsClosed` (state\_id)
  - If state\_id = 14 (Cancelled) OR 16 (Completed) set to closed. Otherwise open.
- `CategoryID` (pulled from RM)
- Description
- `ScheduledDate` (scheduled\_start)
- `AssignedCloseDate` (based on state\_id, nullable)
- `PriorityID` (pulled from RM priorities, priority ID)
- `StatusID` (pulled from RM statuses, status ID)
- `PropertyID` (Only if property and `ref_property_id` is supplied)
- `UnitID` (Only if property and `ref_unit_id` is supplied)
- `TenantID` (Only if tenant with `ref_user_id` is supplied)

#### **Previously created WOs**

By default, we only the WO in Rent Manager when the WO moves to the ‘Canceled' or 'Completed’ statuses in Latchel. We do not update the status in Rent Manager when the work order moves to “Forwarded to PM.” These are the WO fields we update for previously created work orders:

- `AssignedCloseDate` (based on state\_id, nullable)
- `IsClosed` (state\_id)
  - If state\_id = 14 (Cancelled) OR 16 (Completed) set to closed. Otherwise open.
- `StatusID` (pulled from RM statuses, status ID)
- `PropertyID`\* (Only if property and `ref_property_id` is supplied)
- `UnitID`\* (Only if property and `ref_unit_id` is supplied)
- `TenantID*` (Only if tenant with `ref_user_id` is supplied)

\*these fields should never change but are technically included in the update to RM

#### **Integration Work Order Update conditions**

A customer can specify if they want WO updates to happen less often than by default. This settings is located  on a PM account details page → Miscellaneous Settings → Integration Work Order Update conditions. This setting is only accessible by admin users.
