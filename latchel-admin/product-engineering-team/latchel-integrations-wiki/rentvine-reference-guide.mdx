## **Activating the integration**

### **Integration Requirements**

#### **<u>1 - Obtain credentials</u>**

To activate Latchel’s Rentvine integration, engineering will need three pieces of data:

1. API Key
2. API Secret
3. customer subdomain. The subdomain is found in the URL of the customer’s Rentvine portal the first bit after `https://`. In the screenshot below the subdomain is `demopm5` .

<Frame>
  ![IT 6](/images/IT-6.png)
</Frame>

To set up an API key:

1. Log into Rentvine
2. Navigate to Settings\>Account\>User, Roles and API Keys

<Frame>
  ![IT 7](/images/IT-7.png)
</Frame>

3. Go to the API tab
4. Click “New API Key”

<Frame>
  ![IT 8](/images/IT-8.png)
</Frame>

5. Fill out
   - name: Latchel
   - role: Administrator

The customer should then copy and paste text (not screenshot) of the returned API Key and API Secret and send it to their Latchel customer success or product support contact.

<Frame>
  ![IT 9](/images/IT-9.png)
</Frame>

#### **<u>2 – Get Property & Resident Data from Rentvine</u>**

**NOTE**: This step is only necessary for existing customers. If you are working with a new customer with no data in Latchel yet, you can skip to step 4.

This step is critical to avoid creating duplicate property and resident records in Latchel.

Request a CSV export of the “Lease Tenant” report (Reports \> Lease Tenants or search for “Tenant”) from the Rentvine Web App

<Frame>
  ![IT 10](/images/IT-10.png)
</Frame>

#### **<u>3 – Submit a #account-update-request</u>**

In this step, the data support team will clean and prepare the customer’s data by assigning “ref\_ids” to each resident and property in Latchel. This will prevent the integration from accidentally creating those records again when we turn the integration on in the next stage.

1. Navigate to the #account-update-request slack channel
2. Go to Workflows--\>Account Update Request
3. Fill out the required information
   - Select “Data Migration” in the task type question
4. Tag @Jordan Caley in the slack workflow or thread after its created
5. Once the data support team is completes the data work, product will prioritize the engineering ticket to complete the final step of this stage
6. Once product confirms this step is complete, move to the next stage

#### **<u>4 – Notify Engineering</u>**

You’re almost there! The last stage is to notify engineering to activate the integration. Make sure you are provided or have already provided API keys as outlined earlier.

### **What happens when a customer first activates their integration?**

When an admin first activates the customer’s Rentvine integration nothing happens immediately. The integration will be active and will be scheduled for hourly imports but currently does not have an “initial sync” like Propertyware and Rent Manager.

## **Rentvine → Latchel**

The following Property fields are brought over from Rentvine to Latchel.

| **Rentvine Field** | **Value/Interpretation in Latchel** |
| :-- | :-- |
| PropertyId | ref\_property\_id |
| UnitId | ref\_unit\_id |
| address | address1 |
| address2 | address2 |

| stateID | state |
| postcalCode | zip |
| blank | criticalNote |
| unit.isActive | property.is\_active |
| name: [unit.name](http://unit.name) | integration\_data |

#### **Automatic Property Reconciliation**

To ensure accurate customer billing Latchel runs on automated process each month to de-activate any active units in Latchel that no longer exist in Rentvine. This process currently happens on the 27th at 3am PST.

#### **Common areas**

Upon customer request, Latchel can create common area units for PM’s to manage building-level maintenance for jobs such as landscaping, garbage removal, elevator maintenance and more.

How it works:

1. A PM (or Latchel team member on behalf of a PM) submits a one-off request for common area maintenance on a specific building(s) via this Typeform: [https://latchel.typeform.com/to/P8jj3FBx](https://latchel.typeform.com/to/P8jj3FBx)
   - Why one-off requests only? We only want to enable common areas on specific addresses for customers who are actually going to create building-level work orders due to billing implications (they will appear as active units in Latchel for billing purposes).
2. Latchel creates the common area unit via API.
   - This results in a record with a building ID but no ref unit ID.

### **Residents**

The following resident fields are brought over from Rentvine to Latchel. Currently we sync all eligible residents on every sync. We sync all residents on the property so that any of them are able to create a work order in Latchel.

| **Rentvine Field** | **Value/Interpretation in Latchel** |
| :-- | :-- |
| contactID | ref\_user\_id |
| name | name |
| email | email |
| phone | Primary phone number\* |
| propertyId | ref\_property\_id |
| unitId | ref\_unit\_id |
| Move In Date | move\_in\_date |
| Move Out Date | move\_out\_date |

### **Previously-created residents**

Once a resident is created we only change a few things on subsequent syncs

- Property/unit associations
- Move in / move out dates
  - When a resident is passed their move-out date, we’ll mark them inactive at the relevant property in Latchel

### **Work orders**

By default, work orders are not imported from Rentvine to Latchel. If CS or PS receive a specific customer request to activate work order imports:

1. Confirm with the the customer if they want all historic work orders imported to Latchel, or just future work orders.
2. Submit a #feature-request specifying whether the PM wants historic imports or just new work orders, and tag @Jordan Caley and @Ben Heller in the thread.

| **Platform field** | **Field/Interpretation in Latchel** |
| :-- | :-- |
| Work order Number | order\_number |
| Id | ref\_job\_id |
| Description (first 50 characters) | name |
| Description | Description |
| scheduledStartDate | scheduled\_start |
| Status\* | If the work order is vacant, set to “Scheduling with Vendor,” otherwise set  to “Diagnosing Issue” \*We don’t take the platform’s status into account when deciding which Latchel status to set when the WO is created. See rules above. |
| Priority | is\_urgent is true if Priority is “High” (3) |
| ContactID (tenant ID) | ref\_user\_id |

## **Latchel → Rentvine**

### **Work orders**

Work orders are synced to Rentvine a few seconds after they are created in Latchel.

| **Field in Latchel** | **Field/Interpretation in Platform** |
| :-- | :-- |
| order\_number | Order number is sync’d back to Latchel and updated in Latchel to reflect the order number in Rentvine |
| name \+ description | description |
| vendor\_instructions | VendorInstructions |
| isSharedWithTenant | EntryNotes |
| isSharedWithOwner | VendorId Latchel vendor (we create and add a “Latchel Vendor” for this purpose) |
| priorityID | if is\_emergency: 3<br />else 2 |
| workOrderStatusID | Default = Open<br /><br />However, Latchel admins can update this on the PM’s behalf via the Integrations tab \> Initial Work Order status to any of the following options: Open, Completed, Cancelled, Pending, On Hold |
| isVacant | is\_vacant |
| requestedByContactID | tenant\_id |
| ref\_property\_id | propertyID |
| ref\_unit\_id | unitID |

<Note>
  **Rentvine Property Assignee (assignedToUserID) - Available upon request**

  When work orders sync from Latchel to Rentvine, the Rentvine work order property assignee is updated based on the Rentvine lease assignee. To request for a specific PM, please submit a #feature-request and tag @Jordan Caley.
</Note>

### **Previously created WOs**

By default, we only update the WO in Rentvine when the WO moves to the ‘Canceled' or 'Completed’ statuses in Latchel. We do not update the status in Rentvine when the work order moves to “Forwarded to PM.”

### **Work Order Update conditions**

A customer can specify if they want WO updates to happen less often than by default. This settings is located  on a PM company page → Integrations tab → Work Order Update conditions. This setting is only accessible by admin users.
