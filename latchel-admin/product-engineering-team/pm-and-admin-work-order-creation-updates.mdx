## What is the Product Release?

We have simplified the work order creation workflow for PMs and Admins so that they are only asked necessary questions. We have also applied updated designs to this workflow. The biggest change with this update is that we removed the diagnostic workflow.

<Frame>
  ![Work order creation workflow overview](/images/Eng-184.png)
</Frame>

## What was the Previous State?

The PM and Admin work order creation flow had a lot of unnecessary steps and questions that were never really used. There have been many escalations to simplify the workflow so that users can save time when creating a work order.

## Who Is Impacted?

The following groups are impacted:

- PMs
- Latchel Admins/Ops - If Ops needs the diagnostic workflow, they can use the call-center flow.

## When Is It Relevant?

These changes apply to the WO creation workflow for PMs and Latchel Admins on the PM dashboard page and /work-orders page.

<Frame>

This form consists of the following:

- **Work Order Name** - Title for the work order.
- **Work Order Number \[OPTIONAL\]** - Optional field for typing in your own work order number to identify the work order. If you don't fill this out, Latchel will automatically assign a work order number once it's created.
- **Description** - Description to explain the issue that the work order is for. This is important now that the diagnostic workflow has been removed from this workflow.
- **Skill Required** - Choose the skill category and specific skill for this work order. This is what drives service provider assignment and dispatch.
  - You also have the option to press "Ask resident to identify issue?". When you click this option, we will not assign the work order a skill to start and instead send an SMS and Email notification to the resident and ask them to fill out the diagnostic workflow themselves. Once they fill it out, the work order will be updated with the problem, location, and skill and then the work order will go through any corresponding automation.

</Frame>

<Frame>

- Please note that if you change your mind about asking the resident to identify the issue, you can press the "Assign skill manually" option to then select the skill yourself.

<Frame>
  ![Assign skill manually option](/images/Eng-198.png)
</Frame>

- **Max Budget** - Budget that you'd like to set for this work order. The field is automatically populated with the property's soft budget limit on default. We have also included the property's hard budget limit in the subheader to better inform your decision.
- After filling out the Basic Details and then pressing "Next Step", you will then be prompted to fill out **Additional Details**.

<Frame>
  ![Additional Details form](/images/Eng-199.png)
</Frame>

This form consists of the following:

- **Update Resident Availability** - This toggle should be used if the user has the resident's availability to fill out. It will be "No" on default, but switching this toggle to "Yes" will give them an option to choose resident availability. They can also choose pre-set or custom time slots.

<Frame>
  ![Update Resident Availability toggle](/images/Eng-200.png)
</Frame>

- **Can resident cancel this work order?** - This toggle is set to "Yes" by default and should be changed if the user wants to prevent residents from being able to cancel this work order.
- **Does the service provider have permission to enter if the resident is not present?** - If the user has assigned a primary resident to the work order earlier in the workflow, this toggle will be set to "No" on default. The user may change the toggle to "Yes" if they'd like to give the service provider permission to enter. This toggle will also be set to "Yes" on default, if the user has chosen to mark the property as vacant. When this toggle is set to "Yes", it will be mandatory for the user to fill out property access instructions.

<Frame>
  ![Service provider permission to enter toggle](/images/Eng-201.png)
</Frame>

- **Assign a Service Provider manually** - The user should change this toggle to "Yes" if they'd like to manually assign a service provider to the work order rather than have it go through auto assign. The user will have the option to manually add a service provider once the work order is created.
- **Do you have any files related to the work order?** - The user should press "Select File(s)" if they have any files or photos to attach to the work order.
- **Instructions for Service Provider** - This is a mandatory field intended for any special instructions for the service provider.
- After filling out the Additional Details and then pressing "Next Step", you will then be taken to a **Review Work Order** page to look over all your choices before submitting and creating the work order.

<Frame>
  ![Review Work Order page](/images/Eng-202.png)
</Frame>

- If all the fields look good in the Review Work Order page, press "Submit Work Order" in the top right. Then you will be taken to the newly created work order page.
- Please note that the side panel allows you to move amongst the different sections of the work order creation workflow.
  - Anything filled that is bolded and has a black circle next to it means that the section has been completed.
  - The "less than sign" denotes which page you are currently on.

## What Are Some Risks or Ways Things Could Go Wrong?

One of the biggest changes that we are making with this deployment is the removal of choosing the PLC while creating a WO. While this does not impact PMs, there was a call out that Ops may have certain situations where they may need to create a WO to assign a PLC that doesn't exist in the diagnostic workflow (through the call center flow). However, our research showed that 4% of all WOs are being created through the internal web form by Latchel Ops/Admins. From there, 15.7% of those WOs did not get created with the diagnostic workflow (either by the admin in the wo creation workflow or by the resident during diagnosing issue). Then, 7.7% of those did have a PLC assigned to them. This is 0.04% of all WOs. I shared this list of WOs with Ops to validate if they actually could not be created through the diagnostic workflow in the call-center flow and they confirmed that they could have been.

Since our data shows that we don't really need Latchel Ops/Admins to choose a PLC, we decided to move forward without supporting adding a PLC for P0. We will instead closely watch out for this after we deploy and make a decision on if we need to support the custom PLC functionality for Latchel Ops.
