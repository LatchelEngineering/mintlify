## What is the Product Release?

We have updated the Troubleshooting workflow so that Troubleshooters can change the problem, location, or skill for the work order. Our system will then automatically evaluate any updated Problem/Location Combination (PLC) against the PM's severity settings to then determine what the new severity should be for the WO. Please note that this change is only for Emergency WOs

## What was the Previous State?

Previously, we did not have any way to change the problem or location on a WO. As a result, Troubleshooters would try to figure out what the actual PLC is, manually look up what the PM's corresponding setting is for this new PLC, and determine whether to keep the WO as an emergency or de-escalate it. This was also problematic because there was no record of what PLC the Troubleshooter decided to use to determine if the WO is still an emergency or not. We have received escalations in the past from PMs about how Troubleshooters have not provided clear enough notes on why a WO was de-escalated.

## Who Is Impacted?

The following groups will be impacted:

- Troubleshooters will have to learn a new troubleshooting workflow and how to choose the right problem, location, and/or skill.
- Property Managers and Residents will see the following benefits:
  - WOs will now display the proper PLC and/or skill right after the Troubleshooting state.
  - We are mechanizing choosing the corresponding severity setting for the updating PLC.
  - For SaaS customers, we've made sure that de-escalated WOs go through the proper automation (clean up from SaaS hangover).

## When Is It Relevant?

This will only impact WOs that first come in as Emergencies and Troubleshooters will only be able to change problem or location during the Troubleshooting state. They will still have the opportunity to change the skill at any time on the WO page.

### Activity Log Changes

With the introduction of this new Troubleshooting workflow, we've added activity log changes to ensure PMs are able to easily understand the changes that were made during the Troubleshooting process.

#### Updating Problem, Location, and/or Skill

When a problem, location, and/or skill is updated during the Troubleshooting workflow, they will be logged in the activity log like this. Note that they will only show if you make a change.

<Frame>
  ![Activity log entry for updated PLC](/images/Eng-85.png)
</Frame>

If you expand it, you can see what the problem, location, and/or skill was set to previously.

<Frame>
  ![Expanded activity log entry showing previous values](/images/Eng-86.png)

#### Work Order Severity Change

If the WO is de-escalated from an Emergency, then it will add a log item indicating the severity change. Please note that the only two options here are decreasing from Emergency to Normal or Urgent. While we will change a WO to Needs Approval or Resident Responsibility, those technically aren't severities. When a WO Needs Approval or Resident Responsibility, the severity is actually Normal.

In addition, any time a work order is de-escalated from an Emergency, we will create this automated log item indicating that it was de-escalated due to a Troubleshooter changing the problem and/or location during the Troubleshooting workflow and automatically checked against the PM's severity settings.

When you expand the log item, it'll summarize what items were changed (i.e. problem, location, and/or skill) and what it was previously. The screenshot below only includes problem and skill because those were the only things that were changed. This indicates that the location was left alone.

## What Are Some Risks or Ways Things Could Go Wrong?

We don't anticipate too much risk around the feature itself. However, the introduction of this new workflow requires Troubleshooters to know how to choose the right problem, location, and/or skill without the use of a diagnostic workflow. This is a new concept and relies on the Troubleshooter really understanding the Resident's issue and relaying it in the work order. Other than Resident Responsible and Needs Approval, de-escalations will lead to automated dispatch. If we get it wrong, then it could cost us an unnecessary Service Provider dispatch fee.
