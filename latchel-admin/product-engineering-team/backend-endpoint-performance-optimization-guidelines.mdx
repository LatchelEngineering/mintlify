In much of our application we have duplicate running queries. These can come from a variety of sources such as

- model appends
- other methods/configs in the models that are overly hungry loading data
  - ie PmCompanySettings always is queried when a PmCompany model is instantiated
- resources lazy loading relationships

## Diagnosing endpoints that need performance

Right now we are looking to target decreasing latency, decreasing duplicate/unnecessary queries and decrease RDS usage. We believe the lowest hanging fruit with be endpoints with a high total compute time as well as higher average or long tail latency.

[https://latchel.atlassian.net/jira/software/c/projects/LA/boards/3?selectedIssue=LA-6455&sprints=710](https://latchel.atlassian.net/jira/software/c/projects/LA/boards/3?selectedIssue=LA-6455&sprints=710)

## Logging queries

Using performance benchmarks on staging/local is not going to be particularly helpful. Our performance is a complicated relationship between the number of queries we are doing and the size of our database. Instead a helpful tool for chasing these duplicate/unnecessary queries is to log them on your local env.

Place this code inside the boot method of the `AppServiceProvider` to log all database queries

```text
B::listen(function($query) {
    \Log::info(
        $query->sql,
        $query->bindings,
        $query->time
    );
});
```

If you are having trouble finding the source of the queries, a tool like [Clockwork](https://github.com/itsgoingd/clockwork) can be helpful. I've used it by running the install and examining its raw JSON files that get output to the storage folder.

This will log all queries as well as tracelogs.
