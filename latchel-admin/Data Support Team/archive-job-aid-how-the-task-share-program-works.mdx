# Objective

**Purpose**

The purpose of this document is to explain what the "Task Share" Program is and how it works. 

**What is the Task Share Program?**

This program aims at handling all data entry tasks for Property Management companies during launch and throughout their entire journey with Latchel. We will leverage Customer Success Assistant team members to support Latchel Customer Success Managers and Launch Managers with executing data entry tasks.

In order to do so, requests will be created in the ["Task Share" workspace in Jira](https://latchel.atlassian.net/jira/software/projects/TS/boards/12/backlog), and the Task Share Program Manager will prioritize and assign across the team. The Program Manager will monitor and report out on tasks completion while working with the Management team on building the plan (and making adjustments due to changes in bandwidth) and communicating expectations back to the requester.

# Content

## Step-by-Step Guide

## **Process Overview**

1. The requester will create tickets in the [Task Share Jira Workspace](https://latchel.atlassian.net/jira/software/projects/TS/boards/12/backlog?selectedIssue=TS-3) following the guidelines provided on [Step 1.](#ARCHIVEJob-Aid:HowTheTaskShareProgramWorks-step1)

    1. **Note**: Requester should be mindful that requests will be reviewed and planned on a weekly basis, and that each Sprint will be planned to start the next Monday
    
2. The Program Manager will create a "Sprint" in Jira based on current tasks statuses, next tasks priorities, and resource availability. 

    1. Sprints should be shared by Monday **SOD **in the "**#task-share**" channel that will include the Customer Success Team
    2. Assignees and Requesters should review the plan and reply with the following in Slack, by **Monday EOD. **
    
        1. Agree and approve the Sprint → mark post with the check-mark icon ()
        2. Disagree / Disapprove the Sprint → mark post it with the "X" icon ()
        3. **Note**: If there are questions, they should be asked in Slack and the Program Manager will answer there. If the question wasn't answered or the requester disagrees with the answer, mark it as an "X"
        
    3. If nobody expresses disagreement, the Sprint will be implemented 
    4. If someone disagrees with the sprint and it cannot be resolved via Slack, an ad-hoc "Task Share Sync meeting" will happen and the focus will be on discussing priorities and adjusting the plan as needed. 
    
3. On **Fridays**, the Program Manager will communicate on the "#Task-Share" Slack channel that next week's (sprint) plan was finalized and assigned

    1. The team members assigned a task can use the same channel to ask clarifying questions or ask for help (before or during the sprint).
    2. They will refer to this SOP on how to learn about the task assigned to them (NEED SOP - How to handle a Task assigned to me)

2. **Process Exceptions**:

    1. If an urgent request comes after the sprint is planned, or during the execution of the sprint, the requester will have to reach out to the Customer Success Manager via Slack #task-share channel to justify the need to re-prioritize the Sprint to make this request fit in.
    2. Unless this is an Onboarding Task Request, which will always take priority over other tasks, the Customer Success Manager will challenge the level of urgency to avoid creating unnecessary churn to the rest of the team. If the current sprint has several high priority tasks, the Program Manager will escalate to the COO or CEO  to align on priorities
    3. If the urgency is confirmed, the Program Manager will adjust the Sprint to move lower priority tasks for the next Sprint and will communicate back to the whole team via the #Task Share Team" channel that this change was made

## **Step 2. Plan and Assemble Next Sprint (Program Manager)**

1. On **Fridays/Saturdays**, the management team will update the team availability for the next sprint in this page: [Job-Aid: Team Availability & Skills for Task Share Program](https://latchel.atlassian.net/wiki/pages/createpage.action?spaceKey=CS&title=Job-Aid%3A%20Team%20Availability%20%26%20Skills%20for%20Task%20Share%20Program)
2. On **Mondays & Tuesdays**, the Program Manager will first determine which tasks should be prioritized for the next sprint. To do that:

    1. Calculate the time availability for the next sprint = Team availability for next sprint - Tasks being worked on that will take more than one sprint - Buffer of hours to account for time estimate variability
    
        1. We can start with a buffer of 2hrs, as majority of the resources are in production and their availability will be more volatile. In addition, we don't know how accurate the time estimates are. As we learn, we will adjust the buffer time.
        
    2. Identify the tasks in the backlog with higher priority and calculate the amount of hours required to work on them. 
    3. Compare the time available with the time required to work on the high priority tasks:
    
        1. If Time Available > Time Required, the program manager will identify the tasks next in priority (Medium → Low)  that can be completed with the remaining hours available
        2. If Time Available < Time Required, the program manager will go deeper in the priority tasks to define which ones could be pushed to the next sprint. This will be a judgement call, but see below a few guidelines:
        
            1. Onboarding tasks will always take priority
            2. Tasks related to solving a customer experience issue comes next
            
                1. Note: If we are having a customer experience issue, it should be handled as an escalation and not by this program. The program can support with collecting data in the customer account as long as it is not required immediately. Immediate situations like this, should be handled outside of this program
                
            3. Tasks supporting growth comes next 
            4. If there are questions on how to prioritize, reach out to requesters of tasks that don't seem to have the highest priority to ask about the urgency 
            5. Lastly, if still can't finalize the priority list, present the priority tasks to COO or CEO with the recommendation on what should be de-prioritized and finalize prioritization with one of them  
            
        
    
3. Once you have the list of tasks prioritized for the next step, you will compare the skills required to each task with the skills available in the team. Then allocate team members based on their skill proficiency and availability. This will be an iterative exercise.

    1. To aid in this assessment, you can use the instructions below to build a table:
    
        1. Write the first task that was prioritized with their time requirement on the first row
        2. Use the team skill table ( [Job-Aid: Team Availability & Skills for Task Share Program](https://latchel.atlassian.net/wiki/pages/createpage.action?spaceKey=CS&title=Job-Aid%3A%20Team%20Availability%20%26%20Skills%20for%20Task%20Share%20Program)) to show each team member and the skills they have. Replace "High" with 5, "Medium" with 3, "Low "with 1, "None" with 0
        3. Highlight each skill required for the task
        4. In the "Combined Skill level Score for the Task" column, multiply the skill score for each team member including only the highlighted cells. This will help you to see which team member has the best combination of skills for that task
        5. Once you have built the table for the first task, repeat the same to all other tasks that were prioritized
        
    2. To allocate team members to tasks:
    
        1. First look at tasks that require skills that are "rare" in the team, so you can first guarantee the "rare" resource is allocated to the task that requires it
        2. Then look at the tasks that require more skills and allocate team members that have the highest score, unless their availability has already been taken
        3. There is no perfect solution. It will be a balance between skills and availability. Team members with higher skills will be most utilized. If a team member end up without a task, you can review the project this for a simpler project that this person can use to practice further their skills
        
            1. **ATTENTION**:  Team members working on production tasks shouldn't be assigned tasks with tight timelines or due in the middle of the week as their availability is less predictable.
