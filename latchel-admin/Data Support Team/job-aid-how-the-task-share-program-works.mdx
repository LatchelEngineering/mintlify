##   
**Objective**

The purpose of this document is to explain what the “Task Share” Program is and how it works.

‌

**Who should read this document?**

Customer Success Team & Technical Success Specialists

##   
**What is the Task Share Program?**

This program aims at handling all data entry tasks for Property Management companies during launch and throughout their entire journey with Latchel. We will leverage Technical Support Specialist team members to support Latchel Customer Success Managers and Launch Managers with executing data entry tasks.

To do so, requests will be created in the ["Task Share" workspace](https://latchel.atlassian.net/jira/software/projects/TS/boards/12/backlog) in [Jira](https://latchel.atlassian.net/jira/dashboards/last-visited), and the **Task Share Program Manager** will prioritize and assign them across the team. The Program Manager will monitor and report on task completion while working with the Management team on building the plan (and making adjustments due to changes in bandwidth) and communicating expectations back to the requester.

## **Process Overview**

**1.)** The requester will create tickets in the Task Share Jira Workspace following the steps on [_**How to create a Task Share ticket**_](https://latchel.atlassian.net/wiki/spaces/TST/pages/863141952/Job-Aid%3A+How+The+Task+Share+Program+Works#How-to-Create-a-Task-Share-Ticket) in this job aid_**.**_ Once the ticket is posted, a notification is sent via the “#task-share” slack channel. The assigned team member to the ticket must acknowledge receipt of the ticket.

![](blob:https://media.staging.atl-paas.net/?type=file&localId=null&id=5ce89779-536b-4990-af85-8d2a249b98e0&&collection=contentId-863141952&height=602&occurrenceKey=null&width=949&__contextId=null&__displayType=null&__external=false&__fileMimeType=null&__fileName=null&__fileSize=null&__mediaTraceId=null&url=null)
**2.)** The Program Manager will create a “Sprint” in Jira and the plan is based on the current tasks statuses, issue types, tasks priorities, and resources available every Monday. Sprints should be shared by Monday EOD in the “#task-share” channel.

a.) Assignees and requesters should review the Sprint and reply within the “#task-share” channel by Tuesday EOD if they agree with the current Sprint plan.

b.) For any requests that need immediate attention and completion, requesters must post a message in the #task-share Slack channel indicating the reason why the task should be prioritized. The assignee must acknowledge upon receiving the message and make sure to execute the task over other tasks.

![](blob:https://media.staging.atl-paas.net/?type=file&localId=null&id=327974d3-03f2-4686-a6a4-a7b969004cf2&&collection=contentId-863141952&height=345&occurrenceKey=null&width=960&__contextId=null&__displayType=null&__external=false&__fileMimeType=null&__fileName=null&__fileSize=null&__mediaTraceId=null&url=null)
c.) For any requests which need additional information and clarification, the assignee of the task must post a comment in the Jira ticket to ask clarifying questions or ask for help (before or during the sprint). The status of the task must be changed to “Need Help”. The requester must acknowledge and reply upon receiving the comment. If immediate attention is needed, the assignee can also send a private message directly to the requester via slack. 

![](blob:https://media.staging.atl-paas.net/?type=file&localId=null&id=2bcfffcd-bf2b-4e16-8734-9f76e8e508cd&&collection=contentId-863141952&height=552&occurrenceKey=null&width=1600&__contextId=null&__displayType=null&__external=false&__fileMimeType=null&__fileName=null&__fileSize=null&__mediaTraceId=null&url=null)
**3.)** A Weekly Technical Support Review meeting will be held every Wednesday to talk about the tasks in the current Sprint discuss the assignees’ blockers, and the urgency of the tasks posted.

**4.)** On Fridays, the Program Manager will pre-plan the Sprint for next week with the remaining open tasks.

a.) For tasks that need to have a “peer review” or “audit phase”, the assignee will be informed by the auditor that an audit is needed and must finish the audit by Friday EOD. If the audit is not completed, the Program Manager must be informed. The task will then be added back to the Sprint for next week.

### **Process Exceptions**

* If an urgent request comes after the sprint is planned, or during the execution of the sprint, the requester will have to reach out to the Customer Success Manager via the Slack #task-share channel to justify the need to re-prioritize the Sprint to make this request fit in.
* Unless it is an Onboarding Task Request, which will always take priority over other tasks, the Customer Success Manager will challenge the level of urgency to avoid creating unnecessary churn for the rest of the team. If the current sprint has several high-priority tasks, the Program Manager will conduct an emergency meeting or discuss it during the Weekly Success Review to align priorities. 
* If the urgency is confirmed, the Program Manager will adjust the Sprint to move lower priority tasks for the next Sprint and will communicate back to the whole team via the “#Task-Share'' channel that this change was made. Projects deprioritized will be moved to the next sprint, or back to the backlog, depending on priorities.
* Refer to [_**Service-Impacting Tasks or Critical Tasks**_](https://latchel.atlassian.net/wiki/spaces/TST/pages/863141952/Job-Aid%3A+How+The+Task+Share+Program+Works#Service-Impacting-Tasks-or-Critical-Tasks) for more information on these possible exceptions.

## **How to Create a Task Share Ticket**

**STEP A.)** To create the task request, go to the Task Share page and click “Create Issue”.

![](blob:https://media.staging.atl-paas.net/?type=file&localId=null&id=c74b25a2-c16a-4a60-922b-5028a89722fb&&collection=contentId-863141952&height=506&occurrenceKey=null&width=879&__contextId=null&__displayType=null&__external=false&__fileMimeType=null&__fileName=null&__fileSize=null&__mediaTraceId=null&url=null)
**STEP B.)** On the Create Issue page, choose “Task Share (TS)” as the project type. Then select the appropriate “Issue Type” which will be used to allocate resources to each product.

![](blob:https://media.staging.atl-paas.net/?type=file&localId=null&id=5152ba06-e411-44e6-927e-821c5255aaa2&&collection=contentId-863141952&height=255&occurrenceKey=null&width=240&__contextId=null&__displayType=null&__external=false&__fileMimeType=null&__fileName=null&__fileSize=null&__mediaTraceId=null&url=null)
**Issue Types:** 

i. Account Update Tasks - Use this Issue Type to categorize account-updating related tasks of any existing customers at any level of their account. The tasks may include but are not limited to, property and resident updates, vendor prioritization, assign owner, and portfolio management. 

Sample Jira Ticket: <custom data-type="smartlink" data-id="id-0">https://latchel.atlassian.net/browse/TS-1381</custom> 

ii. Onboarding Tasks - Use this Issue Type to categorize any onboarding-related tasks. These tasks will also be flagged and marked as "Highest Priority".

**STEP C.)** In the Create Issue page, type the title of the request in the Summary following the prescribed titles and format in the [_**STANDARDIZED TASK TITLE**_](https://latchel.atlassian.net/wiki/spaces/CS/pages/2485321730/Task+Share+Program+Standardized+Task+Titles) job-aid.

**Example:** “Add Properties/Residents – (Company Level) - GC Realty”

**STEP D.)** In the description box, the requester must lay out all the vital information needed as simply and as easily as possible to perform the task.

**Please be guided by the following questions asked in the Description box:**

i. **What needs to be done?** – you may use the description of the task stated in the [_**STANDARDIZED TASK TITLE**_](https://latchel.atlassian.net/wiki/spaces/CS/pages/2485321730/Task+Share+Program+Standardized+Task+Titles) job-aid. You may also provide additional context on what is being entered, updated, configured, etc. 

ii. **Paste the URL to where the task needs to be executed.**  – paste the link to where the task needs to be executed. This may be a company link, a portfolio link, a vendor page, or a property URL. 

iii. **How should this be executed? Link to SOP or How-to video. –** Tasks should be submitted with a supporting SOP and/or a screen video recording explaining how to perform the task. 

**Note:** 

a.) The SOP should follow the standard Latchel Format: [How to: Create a Latchel SOP](https://latchel.atlassian.net/wiki/spaces/OPS/pages/770115436). 

b.) The execution of the task presented in the screen video recording must be straightforward and comprehensive. If the screen video recording is saved in Google Drive or any cloud-based drive, make sure to share the file with the assignee's Latchel email address. 

c.) Tasks must not be created without an attached supporting SOP and/or screen video recording.

iv. **Report Attachment -** if the attachment or report to be used for the task is saved in a cloud-based drive, please copy and paste the link attachment in this field. Make sure the attachment is shared with the assignee’s Latchel email address.

v. **Related Zendesk Ticket** - _this may only be applicable for Account Update Tasks Issue Type_. Paste the related Zendesk ticket if the Account update is requested via Zendesk.

## **Service-Impacting Tasks or Critical Tasks**

Service-affecting tasks are identified as tasks that can impact a portion or entire operations within Latchel and/or the customer’s company if a relevant task is not executed promptly. These tasks take precedence over all other tasks including Onboarding. These tasks must be accomplished as quickly as possible to avoid any further delays in the operation and/or before the situation becomes even more aggravating for the customer.

**Note:** The reporter must post a ticket that describes why the task needs immediate attention and how it will affect the entire operation if left disregarded. The ticket must be tagged as **“CRITICAL”** in the summary field as shown below.

![](blob:https://media.staging.atl-paas.net/?type=file&localId=null&id=508d8995-8dc2-4f95-9ef5-ca656bc5d202&&collection=contentId-863141952&height=381&occurrenceKey=null&width=612&__contextId=null&__displayType=null&__external=false&__fileMimeType=null&__fileName=null&__fileSize=null&__mediaTraceId=null&url=null)
The requester must inform the program manager of the urgency of the task. The program manager must assign the ticket to a member of the team with only a few assigned tasks. The requester must inform the assignee or the program manager in the #task-share Slack channel instantly or create a meeting with the team member to make sure the task will be executed right away.

**Below are the identified possible service-impacting or critical tasks but not limited to:**

* Vendor Assignment task if the work orders are not assigned to the company's preferred vendor as indicated during their Onboarding.
* Emergency work orders that are not handled as prescribed because of incorrect Severity and Responsibility Settings.
* Properties that are not assigned to the correct Property Managers for escalations.
* Incorrect Escalation contacts
* Owners not receiving maintenance notifications
* Any configuration requests from Property Managers who are tagged on critical based on the number of unsatisfactory ratings and threats to discontinue service.

## **The Task Share Ticket Workflow**

![](blob:https://media.staging.atl-paas.net/?type=file&localId=null&id=3748d7b0-f8c7-486b-acec-24b9ccd9ab6e&&collection=contentId-863141952&height=1249&occurrenceKey=null&width=1132&__contextId=null&__displayType=null&__external=false&__fileMimeType=null&__fileName=null&__fileSize=null&__mediaTraceId=null&url=null)
* Upon Ticket Creation, the ticket will be set as a TO-DO task which will be assigned to any Technical Support Specialist. If the task is not assigned and added to the active Sprint, the ticket will automatically be assigned to the Program Manager and the current active Sprint. The Program Manager will designate the ticket to a Technical Success Specialist.
* The assigned Technical Support Specialist will change the status from TO DO to IN PROGRESS once execution takes place. 
* Once the specialist is done, the ticket must be set to TASK DONE with proper notes placed in the comment section of the Jira ticket.
* Once the ticket is set to TASK DONE, it will transition to the AUDIT PHASE, the status will change to TO-DO AUDIT and the ticket will be re-assigned to the Program Manager.  The Program Manager will take over and the ticket will be placed to AUDIT IN PROGRESS once auditing takes place.

‌

### The In Progress Phase

During this phase, it is expected that the assigned Technical Support Specialist will start executing what is required to fulfill the task. Once the task is turned into IN PROGRESS, the assignee MUST do the following:

**a.)** Upon setting the status of the task to IN PROGRESS, the assignee must add the TIME he/she started working on the task by placing it in the comment section. 

                **Example**: “Time start: 09:30 AM PST”

**b.)** Create the “**worksheet**” for the task, and save it in the specific folder of the customer in the “[_**Property Managers Files**_](https://drive.google.com/drive/u/1/folders/1K-OxCEHlyfI6GNhukN9aNPsZho1yzz7T)” drive. Immediately link the worksheet in the comment section of the Jira ticket. Remember to rename the worksheet such as the sample below:

               **Worksheet File Name Format:** “Worksheet - (Jira Ticket) - (Company Name)”

‌               **Example:** “Worksheet – TS-1284 – Excalibur Homes.csv”

![](blob:https://media.staging.atl-paas.net/?type=file&localId=null&id=db28d7a8-d530-417e-8271-05544a71625d&&collection=contentId-863141952&height=621&occurrenceKey=null&width=1580&__contextId=null&__displayType=null&__external=false&__fileMimeType=null&__fileName=null&__fileSize=null&__mediaTraceId=null&url=null)This figure shows an example Task Share folder where worksheets are stored for a customer.
**c.)** Continue working on the task until completed. Make sure that updates or the progress of the task are well-documented by placing a comment in the Jira task. 

**d.)** Keep TIME TRACKING up to date every day. The assignee must log the hours each time he/she worked on a task. Then document the working hours in the comment section. 

‌              **Example:** “Time worked today: 09:30 AM PST - 12:30 PM PST. “

**NOTE: ** The total hours worked in the TIME TRACKING of the Jira ticket must be the same as the time indicated in the comment section. THIS SHOULD BE DONE EVERY DAY.

**e.)** Once the task is completed, set the ticket to “TASK DONE” to proceed to the “AUDIT PHASE”. Make sure that all updates made are with proper documentation in the comment section of the ticket. 

### The Audit Phase

The purpose of the Audit Transition/Phase is to ensure the accuracy of the configurations and updates made by the assigned Technical Support Specialist on the customer’s account on every task. This will also prevent possible escalations or any pertinent issues that could affect the entire Latchel operations and processes.

As specified in the IN PROGRESS phase, the assignee must have created a worksheet for the task. This worksheet will be used to evaluate the accuracy of the work executed by the Technical Support Specialist. The Program Manager is responsible to audit each of the tasks done by the specialist.

To know more about how the specialist’s quality of work is audited, please check [Calibration Guidelines: Task Share Audit](https://latchel.atlassian.net/wiki/spaces/TST/pages/2650800157) job-aid.

Once the Program Manager finishes the audit, the task should be tagged as “TICKET COMPLETE”, officially closing the Jira Ticket in the current active Sprint. 

Following the [Calibration Guidelines: Task Share Audit](https://latchel.atlassian.net/wiki/spaces/TST/pages/2650800157) job-aid, the Program Manager must provide feedback to the Technical Success Specialist on whether there are misses in the task or not, and the score achieved from the audit.

**Note:** A notification is sent to the Slack channel [#task-share](https://latchel.slack.com/archives/C01961QAWDB) each time a ticket is created, and if the ticket is transitioned to TO-DO AUDIT, and TICKET COMPLETE.
