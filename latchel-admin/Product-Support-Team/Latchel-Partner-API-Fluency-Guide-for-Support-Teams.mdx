# Part 1: How to Use This Guide

This guide has two jobs. First, it teaches you the core concepts behind APIs so you can understand what customers are asking. Second, it gives you a complete reference for Latchel’s Partner API (PAPI) so you can look up the right answer quickly.

By the end, you should be able to:

- Explain what an API is and how it works using non-technical language
- Navigate the Latchel PAPI documentation yourself at [papi.latchel.com](https://papi.latchel.com/api-docs/papi/)
- Identify which endpoint a customer needs based on their question
- Answer the most common API questions without escalating
- Know exactly when something genuinely needs an engineer

**How to use:** Read Parts 1–2 once to build your foundation. Use Parts 3–4 as an ongoing lookup reference. Keep Part 5 (the question framework) bookmarked — it’s your step-by-step process for any new scenario.

# Part 2: API 101 — Core Concepts

## 2.1 What is an API?

API stands for Application Programming Interface. The easiest way to think about it: **an API is a waiter at a restaurant.**

You (the customer) want food from the kitchen. You don’t walk into the kitchen yourself — you tell the waiter what you want, the waiter goes to the kitchen, and comes back with your order.

In tech terms: a customer’s internal system (the diner) wants data from Latchel (the kitchen). The customer’s system makes a request to the Latchel API (the waiter), and the API returns the data (delivers the food).

Latchel’s Partner API (PAPI) is how customers can access their Latchel data — vendors, properties, work orders, tenants, invoices, and more — from their own tools and systems.

## 2.2 What is REST and Why Does it Matter?

Latchel’s PAPI is a REST API. This means requests are made over the web using standard HTTP calls — the same technology your browser uses to load a webpage. There are five types of actions (HTTP methods):

| Method | What it does |
| --- | --- |
| **GET** | Read data. Retrieve a list of vendors, look up a specific job, etc. GET requests never change data. |
| **POST** | Create new data. Create a new job, add a new property, etc. |
| **PUT** | Replace / fully update an existing record. |
| **PATCH** | Partially update an existing record — change just one or two fields. |
| **DELETE** | Remove a record permanently. |

**Support tip:** Most customer questions are about GET requests — they want to read data out of Latchel. If a customer asks how to pull a list of something, it’s always a GET.

## 2.3 What is an Endpoint?

An endpoint is a specific URL that performs a specific action. All Latchel endpoints start with: `https://papi.latchel.com/v1/`

Examples:

- `https://papi.latchel.com/v1/vendor-companies` → List all vendors
- `https://papi.latchel.com/v1/jobs` → List all jobs/work orders
- `https://papi.latchel.com/v1/properties/123` → Get details of property ID 123

The `{placeholders}` in documentation (like `/properties/{property_id}`) mean the customer replaces the placeholder with a real ID from their account.

## 2.4 What are Parameters?

- **Query parameters** — appended to the URL after a `?` to filter, sort, or paginate results. Example: `/jobs?status=open&page=2` returns open jobs on page 2.
- **Path parameters** — embedded in the URL to identify a specific record. Example: `/jobs/456` retrieves job number 456.

## 2.5 What is Authentication?

Latchel uses **API Key authentication**. Every request must include an API key in the request header. The key tells Latchel which account the request is coming from.

Important things to know:

- Each property manager generates their own API key from their Latchel dashboard (Account Settings → Integrations)
- Re-generating immediately invalidates the old key — the customer must update any integrations using the old key
- Latchel does **not** currently support scoped (read-only) API keys — all keys have full access
- For API-specific technical issues: [tech@latchel.com](mailto:tech@latchel.com)

## 2.6 Rate Limits

**600 requests per API key, per 10-minute window.** When the limit is hit, the API returns HTTP 429 (Too Many Requests).

| Response Header | What it means |
| --- | --- |
| `x-ratelimit-limit` | Total requests allowed in the window (600) |
| `x-ratelimit-remaining` | Requests still available in the current window |
| `Retry-After` | How many seconds to wait before retrying |
| `X-RateLimit-Reset` | The exact timestamp when the window resets |

**Support tip:** If a customer reports 429 errors, ask how many requests per minute they’re making. Advise them to add delays between batches or spread the sync over time.

## 2.7 Pagination

Most list endpoints return results in pages, not all records at once. If a customer says they’re only seeing partial data, pagination is usually the reason. Details on pagination parameters for each endpoint are in the PAPI docs.

## 2.8 Environments: Production vs. Testing

| Environment | Base URL | Notes |
| --- | --- | --- |
| **Production (live)** | `https://papi.latchel.com/v1/` | Real account data. Use for actual integrations. |
| **Testing / Sandbox** | `https://latchel-demo.com/v1/` | Demo data only. Requires a separate dev API key. Safe for testing. |

## 2.9 Field Inheritance

Some fields are marked `[INHERITABLE]`. They cascade down: **Company → Portfolio → Property**. If a field is set at the Company level, all Portfolios and Properties inherit it automatically unless overridden.

## 3.2 Anatomy of the Docs Page

| Element | What it means |
| --- | --- |
| Left sidebar | A list of all endpoint categories. Click any to jump to it. |
| Colored method badges | Each endpoint shows its HTTP method (GET, POST, etc.) followed by the path. |
| “\+ Show” toggle | Expand an endpoint to see full details: parameters, request body, and example responses. |

## 3.3 Step-by-Step: Looking Up Any Endpoint

1. Go to [papi.latchel.com/api-docs/papi/](https://papi.latchel.com/api-docs/papi/)
2. Identify the data type the customer is asking about (vendors, jobs, properties, tenants, etc.)
3. Click the matching category in the left sidebar
4. Scan the endpoint list. Find the one matching the action (GET for reading, POST for creating, etc.)
5. Click “\+ Show” to expand it
6. Read the Parameters section to answer questions about filters, pagination, required fields
7. Read the Response section to confirm what fields are returned
8. Share the relevant endpoint URL with the customer

**Pro tip:** Use Ctrl\+F on the docs page and search for a keyword (e.g., “status”, “invoice”). The docs are one long scrollable page.

# Part 4: Full Endpoint Reference

Every endpoint category in the Latchel Partner API, with available endpoints and the most common customer question for each area.

## 4.1 Company

Endpoints about the API user’s company account and users within it.

| Method | Endpoint | What it does |
| --- | --- | --- |
| GET | `/me` | Returns details about the authenticated company. Use this to confirm API key is working. |
| GET | `/company/users` | Lists all users within the company account. |

**Common question:** “How can I confirm my API key is working?” — Have them call GET /me. A successful response confirms authentication.

## 4.2 Jobs (Work Orders)

Endpoints for creating, reading, and managing work orders (called Jobs in the API).

| Method | Endpoint | What it does |
| --- | --- | --- |
| GET | `/jobs` | List all jobs. Supports filters (status, date range, property, etc.) and pagination. |
| POST | `/jobs` | Create a new job/work order. |
| GET | `/jobs/{job_id}` | Get full details of a specific job. |
| PUT | `/jobs/{job_id}` | Update an existing job (full replacement). |
| POST | `/jobs/{job_id}/cancel` | Cancel a specific job. |
| POST | `/jobs/{job_id}/approve` | Approve a job (e.g., approve a bid or budget). |
| POST | `/jobs/{job_id}/deny` | Deny a job approval request. |

**Common question:** “Can I pull all our work orders via API?” — Yes, GET /jobs. Use pagination for large accounts and filters to narrow results.

## 4.3 Job Files

## 4.18 Webhooks (Push Notifications — Separate from PAPI)

In addition to the pull-based PAPI endpoints, Latchel supports **Webhooks**. Webhooks are push notifications — Latchel sends data to the customer’s system automatically when something changes.

Supported webhook events:

- Jobs (work order created or updated)
- Portfolios (created or updated)
- Tenants/Residents (created or updated)

Webhook documentation is separate from PAPI and is available in the Help Center (search “Webhooks”).

**Common question:** “Do you support webhooks so we don’t have to poll the API?” — Yes, for Jobs, Portfolios, and Tenants. Direct customers to the webhook docs in the Help Center.

| --- | --- |

| Pull / read one specific record |  |
| Create something new | POST endpoint |
| Update an existing record | PUT (full update) or PATCH (partial update) |

1. Go to [papi.latchel.com/api-docs/papi/](https://papi.latchel.com/api-docs/papi/)
2. Click the matching category in the left sidebar
3. Find the endpoint matching the action
4. Click “\+ Show” to read parameters and response fields
5. Confirm the endpoint returns what the customer needs
6. Share the direct link to that section

## Step 4 — Answer the Common Follow-Ups

| Customer question | Answer |
| --- | --- |
| How do I authenticate? | API key in the request header. Generate at Account Settings → Integrations in the Latchel dashboard. |
| Why am I only getting some records? | Pagination. Tell them to check for a page or per\_page parameter and iterate through pages. |
| Why am I getting 429 errors? | Rate limit hit. Max 600 requests/10 minutes. Add delays between requests. |
| Can I filter to only active/recent records? | Check the “\+ Show” parameters for that specific endpoint — most list endpoints support filters. |
| I need a read-only key | Not available. Latchel does not support scoped API keys. Escalate to CSM if a blocker. |
| Can I test without affecting live data? | Yes. Use latchel-demo.com/v1 with a separate demo API key. |

# Part 6: When to Escalate

## Handle It Yourself

- Any question covered in Parts 4 or 5 of this guide
- Questions about generating or re-generating an API key
- Rate limit questions — explain the limit and advise on pacing
- Pagination questions — point to the docs for that endpoint
- Requests for read-only API keys — explain not available, note to CSM if a blocker
- Questions about Analytics\+ as an alternative to the API

## Escalate to Engineering ([tech@latchel.com](mailto:tech@latchel.com))

- The API is returning incorrect or missing data for the customer’s account
- A customer’s API key is not working after re-generation
- An endpoint described in the docs is not behaving as documented
- The customer is reporting a consistent API error (4xx or 5xx) not explained by rate limits or auth issues
- A customer wants a new API feature (new endpoint, read-only keys, new filter options)
- Data discrepancies between what the API returns and what appears in the dashboard

## Escalate to CSM ([success@latchel.com](mailto:success@latchel.com))

- A customer wants to discuss API access as part of a broader account conversation
- A customer says a missing API feature is causing them to consider churning
- Requests for a demo/sandbox account or separate development API key
