<Note>
  This SOP details the end-to-end process for the Recurring Work Orders Beta Program. It explains how to handle the consolidated workflow where Property Managers can request both new Common Area properties and recurring maintenance in a single submission.

  The process involves capturing data via dynamic form, validating requests in a central Google Sheet, and utilizing Slack alerts to coordinate between the PMs, the Ops team, and the Associate Program Manager.
</Note>

## Step by step

### Step 1: Request & Intake

**a.** The PM fills out the [**Recurring Work Orders Beta Program form**](https://recurring-work-orders.web.app/) to request a new recurring work order.

<Frame>
  ![image-20260123-193520.png](/images/Intg-129.png)
</Frame>

**b.** The form uses conditional logic:

- **Existing Property:** If the property exists, they provide the URL and maintenance details.
- **New Common Area:** If they select "Is this for a common area property? -\> **Yes**" AND "Is the common area property already created? -\> **No**", the form expands to ask for the building's address details.

**c.** Upon submission:

- Maintenance details flow to the [**Recurring Work Order Request**](https://docs.google.com/spreadsheets/d/1S1uIdgEyqSYh9Jqy6E_N-NoT2WA3WG3sZubhkWJiE4U/edit?gid=1730274070#gid=1730274070) tab.
- If a new Common Area Property was requested, address details flow to the [**Common Area Property Requests**](https://docs.google.com/spreadsheets/d/1S1uIdgEyqSYh9Jqy6E_N-NoT2WA3WG3sZubhkWJiE4U/edit?gid=248650263#gid=248650263) tab.

**d.** Upon submission, the PM sees a confirmation message stating the request has been received and the Ops team will review it.

<Frame>
  ![image-20260108-180523.png](/images/Intg-130.png)
</Frame>

<Note>
  **_Note:_** The form is dynamic. Depending on the PM’s answers (e.g., selecting "No" for resident coordination), certain fields (like Resident URL) may be hidden or optional.
</Note>

### Step 2: Ops Team Verification

**a.** A real-time alert is sent to the [#automatic-recurrent-work-orders](https://latchel.slack.com/archives/C09U7LCP44E) Slack channel. Acknowledge the Slack alert with a checkmark reaction (✅).

<Frame>

</Frame>

**b.** **Duplicate Check:** **_<u>The Ops team must verify the request is not a duplicate</u>_**.

1. PMs only need to capture the information once per task. The automation handles the recurrence.
2. Use `CTRL + F` in the **Recurring Work Order Request** tab to search by Address and Work Order Title to ensure it doesn't already exist. **<u>If it already exists please inform Trisha.</u>**

<Warning>
  Check the **Start Date (Column S)** to confirm when the PM requested this work order to be created. Regardless of the date, Step 2 should be completed.

  If the work order is for a future date later than when the notification was triggered, **automatic step 3** and **manual step 4 and 5** from the Ops team will not be completed until the work order is created. For this reason, the Ops team must check the **Recurring Work Order Request** tab daily, specifically **Column Y (Last Run Date)**, to know when to continue with steps 3–5.
</Warning>

### Step 3: Automated Scheduling (n8n)

Once the status is set to "Active," the automation (n8n) takes over:

**a.** **Last Run Date (Column Y):** Automatically fills with the date the work order was created.

**b.** **Next Run Date (Column Z):** Automatically calculates the next scheduled date based on the frequency selected by the PM.

- _Example:_ If Frequency = Weekly and Start Date = 01/08/2026, the Next Run Date will be 01/15/2026.

**c.** **Satisfaction Survey (Column AA):** Updates to "Yes" automatically once the [**survey**](https://latchel.typeform.com/to/F0NjOaBk)\*\* \*\*email is sent (24 hours after WO creation). If the PM user has submitted another recurring work order request, the survey won’t be sent again and this field will be marked as “Already sent.” This is to avoid overwhelming the PM with survey emails.

<Frame>
  ![image-20260108-182823.png](/images/Intg-142.png)
</Frame>

\*\*d. Submission ID (Column AB): \*\*Unique identifier used to link requests with Recurring requests across tabs. This is filled automatically.

### Step 4: Ops moves the work order forward

**a.** By default, work orders are created with the “Need PM Approval” status.

<Frame>
  ![image-20260109-200004.png](/images/Intg-143.png)
</Frame>

**b.** Before you proceed with moving it, go to the details under the **Details tab \> Work Order Budget Limits**, check the **normal soft limit.**

<Frame>
  ![image-20260123-212320.png](/images/Intg-144.png)
</Frame>

**c.** Open the newly created work order, **click on the button Action** located in the top right corner, and **click on Approve or Deny Work**.

<Frame>
  ![image-20260109-200152.png](/images/Intg-145.png)
</Frame>

**d.** A new window will be open, **click on Approve**.

<Frame>
  ![image-20260109-200326.png](/images/Intg-146.png)
</Frame>

**e.** Input the amount you checked previously under the normal soft limit and type on the reason: Recurring Work Order, and **click Submit**.

<Frame>
  ![image-20260123-211933.png](/images/Intg-147.png)
</Frame>

**f.** This will move the work order to the next status in the workflow. From there, you should let the work order follow the flow on its own.

### **Step 5: Ops Confirms Task Completion**

**a.** In the same alert thread received in the [#automatic-recurrent-work-orders](https://latchel.slack.com/archives/C09U7LCP44E) Slack channel, the Ops team member confirms that the manual steps were completed successfully with a brief message. Depending on the situation, you can use one of the following templates:

- Request is a duplicate; escalated to @Trisha
- Manual steps completed, Work order created.
