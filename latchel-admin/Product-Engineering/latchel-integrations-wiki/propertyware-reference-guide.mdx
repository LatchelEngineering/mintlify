## Enterprise Account Restrictions

Propertyware enterprise accounts with multiple "locations" are only supported if all properties the customer wants to sync are in a **single location**.

## What Happens When a Customer First Activates?

When a customer first activates their integration, a full synchronization of all of their properties and residents is triggered automatically. We refer to this as the **“initial sync.”**

## Platform → Latchel Field Mapping

### Properties

The following Property fields are brought over from PW to Latchel via the initial sync:

| Platform field | Field/Interpretation in Latchel |
| --- | --- |
| Address 1 | address 1 |
| Address2 | address2 |
| City | city |
| State | state |
| Zip Code | zip |
| Platform property ID | ref\_property\_id |
| Platform unit ID | ref\_unit\_id |
|  | critical\_note |
| is\_active | is\_active \*If property is “inactive” in the platform, the property will be marked as “inactive “in Latchel |
| (various data) | integration\_data |

<Warning>
  Multifamily buildings: Sometimes (but not always) fictitious units are created for multifamily buildings. If a customer reports this behavior, escalate to the CSM to propose a billing adjustment.
</Warning>

### Residents

For both initial and regular syncs only residents with active leases and included in imports both for initial and ongoing syncs.

The following resident fields are brought over from PW to Latchel via the initial sync:

- ref\_user\_id
- name
- email
- phone (Use “Mobile” if available. If not, “Home”)
- ref\_property\_id
- ref\_unit\_id
- move\_in\_date (check if valid date object, if not, set to null)
- scheduled\_move\_out\_date (check if valid date object, if not, set to null)

<Note>
  This is different than lease\_end\_date in Propertyware.
</Note>

- integration\_data

### Work Orders

Work orders are excluded from all syncs if the owner email does not match the email used to access Propertyware (as supplied when setting up the integration)

The following Work Order (Job) fields are brought over from PW/RM via the Integration:

- ref\_work\_order\_id
- name (special logic applied to format to Latchel)
- work order number
- description
- scheduled\_data
- vendor\_id
- state\_id (special logic applied to format to Latchel)
- is\_urgent (special logic applied to format to Latchel)
- ref\_property\_id
- ref\_unit\_id (special logic applied to format to Latchel)
- ref\_user\_id (special logic applied to format to Latchel)

Each work order is:

- Assigned to a property manager based on the pm\_user id related to the property
- Linked to an existing property & unit in Latchel
- IF a resident id is supplied, the WO will also be assigned to an existing resident in Latchel.

## Leases

Leases are not currently processed as part of the initial sync.

## **What happens after the initial sync?**

There are two directions of data synchronization that occur after the initial sync.

1. Platform → Latchel
2. Latchel → Platform

Subsequent syncs will look for any data that was created or updated **within the last 72 hours.** The syncs run every other hour, starting sometime in the first 5 minutes of the hour and may take between 15-45 minutes to complete.

Subsequent property updates can be organized into two categories (1) Newly created properties and (2) previously created properties

#### **Newly created Properties**

Newly-created properties will sync all the same fields we do in the initial sync. #### **Common areas**

Upon customer request, Latchel can create common area units for PM’s to manage building-level maintenance for jobs such as landscaping, garbage removal, elevator maintenance and more.

How it works:

1. A PM (or Latchel team member on behalf of a PM) submits a one-off request for common area maintenance on a specific building(s) via this Typeform: [https://latchel.typeform.com/to/P8jj3FBx](https://latchel.typeform.com/to/P8jj3FBx)
   - Why one-off requests only? We only want to enable common areas on specific addresses for customers who are actually going to create building-level work orders due to billing implications (they will appear as active units in Latchel for billing purposes).
2. Latchel creates the common area unit via API.
   - This results in a record with a building ID but no ref unit ID.

<Warning>
  Multifamily buildings: Sometimes (but not always) fictitious units are created for multifamily buildings. If a customer reports this behavior, escalate to the CSM to propose a billing adjustment.
</Warning>

### **Work Orders**

Subsequent work order updates can be organized into two categories (1) Newly-created work orders and (2) previously created work orders

#### **Newly-created work orders**

In order for a WO to sync into Latchel, the “Managed By” field **must** be assigned to the Latchel user in PW.

There is no way to automatically have all of the “Managed By” field filled out when a maintenance request is submitted. It has to be manually done by someone on the team. Normally:

- When a resident submits the WO, the “Managed By” field will be blank.
- When a PW user creates the WO, the “Managed By” defaults to the user that created the WO.

There is some customization:

1. Go to your Settings.
2. Go to Customize Maintenance/Maintenance Settings.
3. There should be a “Managed By” option where you can pick the default user (which could be the Latchel user).

Newly-created work orders will sync all the same fields and following the same rules as in the initial sync.

#### **Previously-created work orders**

The integration will not update any fields on previously created work orders

### **Leases**

The lease sync runs at 11:30pm UTC every night and are thus synchronized from PW to Latchel once a day.

The following lease fields are brought over from PW to Latchel via the daily sync:

- `ref_lease_id`
- `ref_tenant_id`
- `ref_property_id`
- `ref_unit_id`
- `integration_data`
- `start_date`
- `end_date`
- `amount`
- `status`
- `term`

#### **Newly-created leases**

Sync’d daily at 11:30pm UTC every night

#### **Previously-created leases**

We update all values if it is an existing lease. So whatever new values come in from the sync would override the old ones.

## **Latchel → Platform**

### **Work orders**

If a work order (job) is created in the Latchel console it will automatically trigger a sync to also create it in PW. Updating that work order (job) in Latchel will automatically sync over the changes from Latchel to PW.

- refWorkOrderId
- work order number
- Type
  - always set to “General” for Propertyware
- Priority
  - Set to “High” if the wo is emergency or urgent, otherwise “Medium”
- Status (Default behavior - see section below for detailed settings)
  - Set to “Closed” if state\_id = 14 (Cancelled) OR 16 (Completed). Otherwise set to “Open”
- Description
- StartDate
- approved (true/false)
  - Always set to “True”
- ApprovedDate
  - Created date or null
- Completed Date
- Location
  - we generate a slug based on the integration\_data, by looking at the building, portfolio and unit slugs
- Specific location
  - same as WO name
- Portfolio information
  - from the portfolio\_id inside the integration\_data
- Building information
  - from the building\_id inside the integration\_data
- Unit IDs

#### **Previously-created work orders**

By default, we only the WO in PropertyWare when the WO moves to the ‘Canceled' or 'Completed’ statuses in Latchel. We do not update the status in Rent Manager when the work order moves to “Forwarded to PM.” These are the WO fields we update for previously created work orders:

- type
- priority
- approved
- specific location
- description
- approved date
- status
- start date
- completed date (if avail)

### **File Sync**

Latchel will automatically sync files (such as pictures and invoices) to Propertyware on any work order state change.

<Note>
  If a files from Latchel are automatically published to both the Owner and Resident portals in Propertyware, this behavior is not caused by Latchel’s sync. It is controlled entirely by settings within the Propertyware account. Steps to disable auto-publishing in Propertyware:

  1. Go to Setup
  2. Expand the Customize section
  3. Click on Maintenance
  4. Click Edit Maintenance Settings
  5. Change the following:\
       a. Publish Work Orders to Tenant Portal → No\
       b. Publish Work Orders to Owner Portal → No
  6. Click Save

  _Important: This change applies across the entire Propertyware account—not just to files synced from Latchel._
</Note>
