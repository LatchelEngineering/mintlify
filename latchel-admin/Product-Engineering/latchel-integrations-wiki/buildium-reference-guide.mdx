### **<u>3 – Submit a #task-share-jira-request</u>**

In this step, the data support team will clean and prepare the customer’s data by assigning “ref\_ids” to each resident and property in Latchel. This will prevent the integration from accidentally creating those records again when we turn the integration on in the next stage.

1. Navigate to the #task-share-jira-request slack channel
2. Go to Workflows--\>Account Update Request
3. Fill out the required information
   1. Select “Data Migration” in the task type question
4. Tag @Jordan Caley in the Slack workflow or thread after it's created
5. Once the data support team completes the data work, product will prioritize the engineering ticket to complete the final step of this stage
6. Once product confirms this step is complete, move to the next stage

### **<u>4 – Notify Engineering</u>**

You’re almost there! The last stage is to notify engineering to activate the integration. Make sure you are provided or have already provided API keys as outlined earlier.

## **What happens when a customer first activates their integration?**

When an admin first activates the customer’s Buildium integration nothing happens immediately. The integration will be active and will be scheduled for hourly imports but currently does not have an “initial sync” like Propertyware and Rent Manager.

### **Previously created Properties**

Subsequent synchronizations of the same property will impact the following fields:

| **Platform field** | **Field/Interpretation in Latchel** |
| :-- | :-- |
| (various data) | integration\_data |
| Address data\* | address\_label \*If any address data is different in the platform than in Latchel, we store the exact platform address in order to provide the “Source Address” feature for customers. \***Note**: In Buildium, customers can mark a property as either active or inactive. That setting applies to all units in the property. So if a customer marks a property with multiple units as inactive in Buildium, all of the Latchel properties (units) will also be marked as inactive in Latchel.

#### **Automatic Property Reconciliation**

To ensure accurate customer billing Latchel runs on automated process each month to de-activate any active units in Latchel that no longer exist in Buildium. This process currently happens on the 27th at 3am PST.

### **Common areas**

Upon customer request, Latchel can create common area units for PM’s to manage building-level maintenance for jobs such as landscaping, garbage removal, elevator maintenance and more.

How it works:

1. A PM (or Latchel team member on behalf of a PM) submits a one-off request for common area maintenance on a specific building(s) via this Typeform: [<u>https://latchel.typeform.com/to/P8jj3FBx</u>](https://latchel.typeform.com/to/P8jj3FBx)
   1. Why one-off requests only? We only want to enable common areas on specific addresses for customers who are actually going to create building-level work orders due to billing implications (they will appear as active units in Latchel for billing purposes).
2. Latchel creates the common area unit via API.
   1. This results in a record with a building ID but no ref unit ID.

## **Residents**

The following resident fields are brought over from Buildium to Latchel. Currently we sync all eligible residents on every sync. This differs slightly from our other integrations where we only sync residents that were added or edited in the last 72 hours. An “eligible” resident is anyone living at the property who is not in the Buildium status `['Past']`. We sync all residents on the property so that any of them are able to create a work order in Latchel.

| **Buildium Field** | **Value/Interpretation in Latchel** |
| :-- | :-- |
| Id | ref\_user\_id |
| Firstname \+ Lastname | name |
| Email | email |
| Phone  -\> phone number prioritize: cell \> home \> work \> fax | Primary phone number\* |
| PropertyId | ref\_property\_id |
| UnitId | ref\_unit\_id |
| Move In Date | move\_in\_date |
| Move Out Date | move\_out\_date |

<Note>
  We need to set a primary phone number for the resident in Latchel. Our current logic states that when multiple numbers are provided, we will prioritize setting the primary phone number in the order listed in the table.
</Note>

### **Previously-created residents**

Once a resident is created we only change a few things on subsequent syncs

- Property/unit associations
- Move in / move out dates
  - When a resident is passed their move-out date, we’ll mark them inactive at the relevant property in Latchel

## **Work orders**

By default, work orders are not imported from Buildium to Latchel. If CS or PS receive a specific customer request to activate work order imports, please submit a #feature-request and tag @Jordan Caley and @Ben Heller in the thread.

If a customer requests to activate work order imports, the Buildium integration syncs work orders that have been created in the last 72 hours.

| **Platform field** | **Field/Interpretation in Latchel** |

| EntryContact-\>Id | ref\_user\_id |

## **Product Support FAQs**

1. **Can you sync the Latchel work order number to match Buildium’s?**

We would love to support this feature!  Unfortunately the Buildium API doesn’t currently allow us to read the Buildium work order number. You can help us convince Buildium to enable this feature by sending this note to your Buildium account representative or submitting a support request:

> “Please allow us to retrieve task and work order numbers from the API. We need this data to be able to visually reference work orders between systems”

 

2. **Can you automatically update the work order/task status in Buildium when the work order is complete in Latchel?**

We would love to support this feature!  Unfortunately the Buildium API doesn’t currently allow us to update the status of a task or work order. You can help us convince Buildium to enable this feature by sending this note to your Buildium account representative or submitting a support request:

> “Please allow us to update the task and work order status via API. We need this functionality to be able to keep work orders updated in Buildium.”

3. **We only want work orders to sync from Buildium to Latchel, not the other direction. How do we do it?**

In Buildium, go to Settings \> Developer Tools and locate the API key created for Latchel and click the three dots to edit.

For the “Maintenance” permission, leave “VIEW” checked but uncheck “EDIT” just like the screenshot below. Thats it! Now work orders will only sync from Buildium to Latchel.
