This document is the detailed source of truth for how the Latchel\<\>Yardi sync operates today.

## **Yardi Version Compatibility Requirements**

In order to use the Latchel-Yardi Integration, customers must have Voyager 7S with the following minimum plug-in versions\*: 

- Interfaces PIv22 
- Common Data PIv19 
- Service Requests PIv15

Clients on earlier plug-in versions can still use Latchel by entering enter a support ticket to their Yardi Account Manager or team to request the vendor package for Latchel to be loaded to their database. This will make Latchel appear in their Voyager environment for configuration. Note that no vendor package will be required after clients install _Interfaces Service Requests PIv21,_ once released, as the Latchel package will be included in that version.

Finally, clients with questions about compatibility between their Voyager 7S or other Yardi software and Latchel should contact a Yardi Account Manager directly.

# **Activating the integration**

There are a few high level “stages” in activating the Yardi integration for customers.

<Note>
  It is critically important that the steps are conducted in the order defined below. Going out of order could cause issues with the customer’s data in Latchel and/or Yardi.
</Note>

### Stage 1 - Yardi Voyager Standard Interfaces Setup

The following steps all occur within the customer’s Yardi.

**1 – Configure Vendor Options for Latchel**

Menu Path: Basic iData Menu \> Interfaces \> Configuration \> Review Interface Vendors \> Interface Configuration

1. Confirm the vendor name appears in Voyager under the appropriate Interface configuration.  If vendor is not present, contact Yardi Technical Support to load the vendor package.
2. Select the vendor name.
3. Select the options desired on the vendor configuration screen.  This is the configuration Latchel recommends:

<Frame>
  ![Image 20231017 230947](/images/image-20231017-230947.png)
</Frame>

4. Save the configuration options!

**2 – Configure the Property or Property List**

_Menu Path for a Single Property:  Setup \> Property \> Find Property \> \[property code\] \> Submit._

1. Click on Functions drop down menu.  Click on Interface Configuration.
2. Select “Service Request” for the Interface Type.
3. Highlight Latchel under Available Agencies. Click Select and Save.

_Menu Path for a Property List:  Interfaces \> Configuration \> Bulk Property Configuration._

1. Enter property list code
2. Select Interface Type from drop down.
3. Select Configure.
4. Highlight Latchel from list of Available Agencies and click Save.  

**3 – Configure the Interface User**

Menu Path: System Administration \> Security \> Add User

1. Create a new interface user and link to a security group for the specific Interface Type. The security group should have Read or Read/Write access for appropriate web services permission(s) 

(e.g.) Interfaces: Web Service – Common Data

2. Enter the property code or property list code in the property field to allow interface user access to properties configured for Latchel.
3. Click on Privileges tab and Select “Interface User”.  This will give user access through the web services, but not the Voyager front end.
4. Save.\*

**4 – Provide Access Info to Latchel**

1. Voyager URL (vendor will append the appropriate web services endpoint)
2. Interface username and password
3. Database Name
4. Server Name (will always be the same as Database Name)

<Note>
  _Clients should always notify the third-party vendor of any changes to Voyager URL, database server or database name._
</Note>

\_\*Mutual Clients should enter a case in Client Central if assistance is needed with any Step in this document.  \_

### Stage 2 — Existing Customer Data Migration

If you’re helping a _new_ customer (no property/resident data in Latchel yet) you can skip to the next stage.  Otherwise read on!

### **<u>1 – Get Property & Resident Data from Yardi</u>**

This step is critical to avoid creating duplicate property and resident records in Latchel.

1. Request all resident and property data for the “property list” that was created in Stage 1, Step 2 above.
2. There are multiple ways to get this data in Yardi. The customer can generate and send the data to us in whatever way is easiest for them. The fields that are most import to us are
   - A. Property
     1. Address 1
     2. Address 2
     3. pCode (unique ID in Yardi)
   - B. Resident
     1. Name
     2. Contact info
     3. move-in dates
     4. move-out dates
     5. tCode (unique ID in Yardi)

### **<u>2 – Submit a #account-update-request</u>**

In this step, the data support team will clean and prepare the customer’s data by assigning “ref\_ids” to each resident and property in Latchel. This will prevent the integration from accidentally creating those records again when we turn the integration on in the next stage.

1. Navigate to the #task-share-jira-request slack channel
2. Go to Workflows--\>Account Update Request
3. Fill out the required information
   1. Select “Data Migration” in the task type question
4. Tag @Jordan Caley  in the slack workflow or thread after its created
5. Once the data support team is completes the data work, product will prioritize the engineering ticket to complete the final step of this stage
6. Once product confirms this step is complete, move to the next stage

### **Stage 3 - Activate the Integration in Latchel**

You’re almost there! The last stage is turning on the integration in Latchel.

### **<u>1 – Gather Customer’s Yardi Info</u>**

The customer should have provided this to you as part of Stage 1, step 4. You will need it to continue.

### **<u>2 – Activate the Integration in Latchel</u>**

1. Go to the customer’s integration page in Latchel e.g. [<u>https://app.latchel.com/admin/pm-company/2/integrations</u>](https://app.latchel.com/admin/pm-company/2/integrations)
2. Find the Yardi tile and click “Activate”
3. Follow the steps in the video below for entering the rest of the information.
4. Thats it! The customer’s Yardi integration is live!

## **What happens when a customer first activates their integration?**

When an admin first activates the customer’s Yardi integration nothing happens immediately. The integration will be active and will be scheduled for hourly imports but currently does not have an “initial sync” like Propertyware and Rent Manager.

### **Common areas**

Upon customer request, Latchel can create common area units for PM’s to manage building-level maintenance for jobs such as landscaping, garbage removal, elevator maintenance and more.

How it works:

1. A PM (or Latchel team member on behalf of a PM) submits a one-off request for common area maintenance on a specific building(s) via this Typeform: [https://latchel.typeform.com/to/P8jj3FBx](https://latchel.typeform.com/to/P8jj3FBx)
   1. Why one-off requests only? We only want to enable common areas on specific addresses for customers who are actually going to create building-level work orders due to billing implications (they will appear as active units in Latchel for billing purposes).
2. Latchel creates the common area unit via API.
   1. This results in a record with a building ID but no ref unit ID.

## **Residents**

The following resident fields are brought over from Yardi to Latchel. Currently we sync all eligible residents on every sync. This differs slightly from our other integrations where we only sync residents that were added or edited in the last 72 hours. An “eligible” resident is anyone living at the property who is not in one of the following Yardi statuses `['Applicant', ‘Denied', ‘Past', 'Canceled']`. We sync all residents on the property (Yardi’s concept of “current” and “roommates”) so that any of them are able to create a work order in Latchel.

| **Latchel Field** | **Value/Interpretation** |
| :-- | :-- |
| ref\_user\_id | @tCode (tenant code) |
| name | firstname \+ lastname |
| email | Email |
| phone | Phone  -\> phone number prioritize: cell \> home \> office\* |
| ref\_property\_id | Property code |
| ref\_unit\_id | Unit code |
| move\_in\_date | moveindate |
| move\_out\_date | moveoutdate |

<Note>
  We need to set a primary phone number for the resident in Latchel. Our current logic states that when multiple numbers are provided, we will prioritize setting the primary phone number in the order listed in the table.
</Note>

### **Previously-created residents**

Once a resident is created we only change a few things on subsequent syncs

- Property/unit associations
- Move in / move out dates
  - When a resident is passed their move-out date, we’ll mark them inactive at the relevant property in Latchel

## **Work orders**

Work orders must be updated in the last 72 hours and be associated with a property in the “property list” Latchel has access to as part of the Yardi integration.

### **Customize which work oders sync from Yardi to Latchel**

By default, all work orders are sync’d to Latchel. However customers can configure exactly how they want work orders to sync. The customer has three options

- **Sync No Work Orders**: This is the default setting and doesn’t require any extra work
- **Sync All Work Orders**: This requires a configuration change from engineering as part of the Yardi set-up process
- **Sync Some Work Orders**: This requires setting up a custom work order field in Yardi and naming it “SyncToLatchel. See [Yardi Reference Guide | Setting Up a Customized Field in Voyager](https://latchel.atlassian.net/wiki/spaces/PROD/pages/2712928302/Yardi+Reference+Guide#Setting-Up-a-Customized-Field-in-Voyager) for enabling this option.

#### **WO Sync Logic**

In general, we default to respecting the custom field on each WO (when it is set) and default to the default setting when its not.

| **SyncToLatchel (on ea. WO)** | **sync\_all\_wo (on PM’s settings)** | **Decision - sync WO to Latchel?** |
| :-- | :-- | :-- |
| left blank or not configured | false | Don’t Sync |
| true | false | Sync |
| false | false | Don’t Sync |
| left blank or not configured | true | Sync |
| false | true | Don’t Sync |
| true | true | Sync |

#### **Setting Up a Customized Field in Voyager**

Unfortunately Yardi does not give us detailed customer-facing instructions on how to set up this field. Yardi’s guidance is for the client to refer to their core Voyager team for assistance modifying these fields. Latchel can also cut a case with Yardi support on the customer’s behalf.

Here are the details we know about setting up the custom field:

- The field must be exactly “SyncToLatchel” (its case sensitive, no trailing spaces etc.)
- The field must be under “General Info”
