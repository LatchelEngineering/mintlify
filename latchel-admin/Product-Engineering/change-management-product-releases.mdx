## Overview

The purpose of this page is to define the Product process for any deployment that requires change management.

We understand that not every product release is large enough to require the processes defined in this page. As a result, we've determined that the process should only be applied for any release where a group of Latchel stakeholders (Ops, PMs, CS, Sales, etc.) needs to change its behavior to enable a better customer experience. If you're unsure on whether or not the Product change requires the process defined here, please make sure to ask someone from the impacted stakeholder team on whether or not it's necessary.

## Change Management and Implementation Process

The table below shows the change management and implementation responsibilities with corresponding owners.

| Responsibility | Owner |
| --- | --- |
| **Pre-Work** |  |
| Conduct product research, analysis, engineering project management, and product testing. | Product |
| **Launch Prep** |  |
| Identify implementation key stakeholders and conducts kick-off review of new product(s). | Product |
| Establish communication channels and cadence (if needed) for ongoing communication. | Product |
| Identify the "Main Stakeholder Team Champion" who is responsible for successful implementation of the product with the impacted group (i.e. CS for PM process change management, Ops for Ops process change management). | Main Stakeholder Team |
| Define scope and scale of initial implementation or pilot. | Jointly Owned (Product & Main Stakeholder Team) |
| Provide set inputs to ensure handoff to the stakeholder team champion. Please see "Set Product Inputs" below for details on the set inputs and templates. | Product |
| Translate set inputs from Product into any training materials (i.e. SOPs, Coassemble, Confluence, Knowledge Base, etc.). | Main Stakeholder Team |
| Ensure launch plan meets implementation goals and any defined prerequisites. | Product |
| Ensure training material is distributed and necessary training is conducted. | Main Stakeholder Team |
| **Launch and Iterate** |  |
| Conduct Go/No-Go meeting (or Slack thread) to confirm all parties are ready. | Product |
| Monitor and measure how the new product performs against business goals. | Product |
| Monitor compliance to implementation plan with impacted group. | Main Stakeholder Team |
| Provide feedback to Product team on iterations and improvements needed. Please see "War Room Post Launch" below for details on where to provide this feedback. | Main Stakeholder Team |
| Iterates towards goals based on feedback and analysis. | Product |
| **Post-Launch Maintenance** |  |
| Uses #product-release channel to announce small feature improvements. | Product |
| Iterates towards goals based on feedback and analysis. | Main Stakeholder Team |

## Set Product Inputs

This section defines the inputs that the Product team will provide to the Main Stakeholder Team Champion so that he/she can properly execute any product launch or change. Product will provide all this information within the [\[Change Management\] Product Releases](/change-management-product-releases) section in Confluence. Each product release will have its own child page under that section. Please go to [this page](/change-management-product-input-template) for an example with a previous product release.

| Set Product Input | Definition |
| --- | --- |
| What is the Product Release? | Summary of what the product release is and why it was designed or built. |
| Who is Impacted? | Summary of who is impacted with this product release. |
| When is it Relevant? | Summary of when the product release is used. |
| What was the Previous State? | Summary of the issue that we're solving and what the state of the world looked like before. |
| How is it Used? | Step by step summary of how to use the new product release. This should include a visual component (i.e. loom, screenshots, etc.). Please ensure you put the link for this section within the top of this section. This is so any stakeholder teams can link to the step by step process easily. |
| What are the Risks? | Not all of our product releases will be perfect in the first iteration and will have some known risks. This will help to prep the impacted group for any of these when deployed. |
| Deployment Date | The date for when the product will be deployed. This will be decided during launch prep between Product and the Main Stakeholder Team Champion, as some releases will require built-in time for training and execution. |
| How to Measure Compliance (optional) | This is optional depending on if measuring compliance is required for success. When it is, Product will provide any necessary information or data on how to monitor compliance for the product release. |

## War Room Post Launch

This section defines how to handle iterations and general post-launch maintenance. We've decided on setting up a war room that allows for any impacted parties to submit any feedback, bugs, or issues. In order to make this successful, we've determined that the information in the table below is required.

| War Room Input | Description |
| --- | --- |
| What to Include? | Product Manager and Main Stakeholder Team Champion to include a brief description of the channel's purpose and pin links to any important documents. |
| Who to Include? | War room channels should always be public. However, as a baseline, add reps from CS, Growth and Ops Leadership. All Product Managers should also be included in war rooms for cross-product knowledge. Department representatives should be responsible for marketing the channel or adding stakeholders as they see fit. |
| When to Take Down? | Product Manager will make the decision on when to take the war room channel down. Generally, if the launch is stable and there is no activity for \> 2 weeks then it may be a good indicator to archive the war room. |
