# Latchel Partner API — Fluency Guide for Support Teams

Internal use only — do not distribute externally. This guide enables the Product Support
team to answer customer API questions independently, without engineering escalation.
Last updated: May 2026.
Part 1: How to Use This Guide
This guide has two jobs. First, it teaches you the core concepts behind APIs so you can
understand what customers are asking. Second, it gives you a complete reference for Latchelʼs
Partner API (PAPI) so you can look up the right answer quickly.
By the end, you should be able to:
Explain what an API is and how it works using non-technical language
Navigate the Latchel PAPI documentation yourself at papi.latchel.com
Identify which endpoint a customer needs based on their question
Answer the most common API questions without escalating
Know exactly when something genuinely needs an engineer
How to use: Read Parts 1–2 once to build your foundation. Use Parts 3–4 as an ongoing
lookup reference. Keep Part 5 (the question framework) bookmarked — itʼs your step-by-
step process for any new scenario.
Part 2: API 101 — Core Concepts
2.1 What is an API?
API stands for Application Programming Interface. The easiest way to think about it: an API is a
waiter at a restaurant.
You (the customer) want food from the kitchen. You donʼt walk into the kitchen yourself — you
tell the waiter what you want, the waiter goes to the kitchen, and comes back with your order.
In tech terms: a customerʼs internal system (the diner) wants data from Latchel (the kitchen).
The customerʼs system makes a request to the Latchel API (the waiter), and the API returns the
data (delivers the food).
Latchelʼs Partner API (PAPI) is how customers can access their Latchel data — vendors,
properties, work orders, tenants, invoices, and more — from their own tools and systems.
2.2 What is REST and Why Does it Matter?
Latchelʼs PAPI is a REST API. This means requests are made over the web using standard HTTP
calls — the same technology your browser uses to load a webpage. There are five types of
actions (HTTP methods):
Method What it does
GET Read data. Retrieve a list of vendors, look up a
specific job, etc. GET requests never change
data.
POST Create new data. Create a new job, add a new
property, etc.
PUT Replace / fully update an existing record.
PATCH Partially update an existing record — change
just one or two fields.
DELETE Remove a record permanently.
Support tip: Most customer questions are about GET requests — they want to read data
out of Latchel. If a customer asks how to pull a list of something, itʼs always a GET.
2.3 What is an Endpoint?
An endpoint is a specific URL that performs a specific action. All Latchel endpoints start with:
https://papi.latchel.com/v1/
Examples:
https://papi.latchel.com/v1/vendor-companies → List all vendors
https://papi.latchel.com/v1/jobs → List all jobs/work orders
https://papi.latchel.com/v1/properties/123 → Get details of property ID 123
The {placeholders} in documentation (like /properties/{property_id} ) mean the
customer replaces the placeholder with a real ID from their account.
2.4 What are Parameters?
Query parameters — appended to the URL after a ? to filter, sort, or paginate results.
Example: /jobs?status=open&page=2 returns open jobs on page 2.
Path parameters — embedded in the URL to identify a specific record. Example:
/jobs/456 retrieves job number 456.
2.5 What is Authentication?
Latchel uses API Key authentication. Every request must include an API key in the request
header. The key tells Latchel which account the request is coming from.
Important things to know:
Each property manager generates their own API key from their Latchel dashboard (Account
Settings → Integrations)
Re-generating immediately invalidates the old key — the customer must update any
integrations using the old key
Latchel does not currently support scoped (read-only) API keys — all keys have full access
For API-specific technical issues: tech@latchel.com
2.6 Rate Limits
600 requests per API key, per 10-minute window. When the limit is hit, the API returns HTTP
429 (Too Many Requests).
Response Header What it means
x-ratelimit-limit Total requests allowed in the window (600)
x-ratelimit-remaining Requests still available in the current window
Retry-After How many seconds to wait before retrying
X-RateLimit-Reset The exact timestamp when the window resets
Support tip: If a customer reports 429 errors, ask how many requests per minute theyʼre
making. Advise them to add delays between batches or spread the sync over time.
2.7 Pagination
Most list endpoints return results in pages, not all records at once. If a customer says theyʼre
only seeing partial data, pagination is usually the reason. Details on pagination parameters for
each endpoint are in the PAPI docs.
2.8 Environments: Production vs. Testing
Environment Base URL Notes

m/v1/ actual integrations.
Testing / Sandbox https://latchel- Demo data only. Requires a

2.9 Field Inheritance
Some fields are marked [INHERITABLE] . They cascade down: Company → Portfolio →
Property. If a field is set at the Company level, all Portfolios and Properties inherit it
automatically unless overridden.

4.18 Webhooks (Push Notifications — Separate from PAPI)
In addition to the pull-based PAPI endpoints, Latchel supports Webhooks. Webhooks are push
notifications — Latchel sends data to the customerʼs system automatically when something
changes.
Supported webhook events:
Jobs (work order created or updated)
Portfolios (created or updated)
Tenants/Residents (created or updated)
Webhook documentation is separate from PAPI and is available in the Help Center (search
“Webhooks”).
Common question: “Do you support webhooks so we donʼt have to poll the API?” — Yes,
for Jobs, Portfolios, and Tenants. Direct customers to the webhook docs in the Help
Center.

Part 6: When to Escalate
Handle It Yourself
Any question covered in Parts 4 or 5 of this guide
Questions about generating or re-generating an API key
Rate limit questions — explain the limit and advise on pacing
Pagination questions — point to the docs for that endpoint
Requests for read-only API keys — explain not available, note to CSM if a blocker
Questions about Analytics+ as an alternative to the API
Escalate to Engineering (tech@latchel.com)
The API is returning incorrect or missing data for the customerʼs account
A customerʼs API key is not working after re-generation
An endpoint described in the docs is not behaving as documented
The customer is reporting a consistent API error (4xx or 5xx) not explained by rate limits or
auth issues
A customer wants a new API feature (new endpoint, read-only keys, new filter options)
Data discrepancies between what the API returns and what appears in the dashboard
Escalate to CSM (success@latchel.com)
A customer wants to discuss API access as part of a broader account conversation
A customer says a missing API feature is causing them to consider churning
Requests for a demo/sandbox account or separate development API key
